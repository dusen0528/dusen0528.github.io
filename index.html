Privacy AI

실시간 얼굴 블러 기능 최초 도입 및 상용화 검증

WebRTC 적응형 비트레이트 환경의 해상도 변동 문제를 해결해 얼굴 블러 기능을 9채널 이상, 24시간 무중단으로 검증한 사례입니다.

WebRTC YOLO FFmpeg OpenCV FastAPI
상세 이력 보기

문제 상황

기존 영상 관제 솔루션에는 AI 기반 개인정보 보호 기능이 없었고, WebRTC 적응형 비트레이트 환경에서는 입력 해상도가 실시간으로 변해 프레임 깨짐, 버퍼 꼬임, 누적 지연이 반복되었습니다. 이 때문에 실시간 얼굴 블러 기능을 안정적으로 서비스하기 어려웠습니다.

해결 방식

  • YOLO 기반 얼굴 검출 및 블러 처리 파이프라인을 설계하고, Python 기반 AI 모듈을 독립 구조로 분리해 특정 연산 병목이 전체 스트림에 전이되지 않도록 구성
  • OpenCV Letterbox 기반 입력 정규화 로직을 적용해 해상도가 변해도 FFmpeg와 AI 모델이 항상 고정 규격 입력을 처리하도록 보정
  • FastAPI 기반 제어 API를 구축해 스트림 상태 모니터링, 설정 변경, 재시작 제어를 런타임에서 수행할 수 있게 개선

성과

9채널 이상 다중 스트림 환경에서 24시간 무중단 운영 검증을 완료했고, 해당 기능은 차기 솔루션 정식 버전에 반영이 확정되었습니다. 실시간 환경의 변동성을 제어하면서 AI 기능을 실제 운영 가능한 구조로 서비스화한 경험입니다.

Audio AI

비명 감지 AI 모델 설계 및 운영 체계 구축

현장 소음과 실서비스 제약을 고려해 모델 설계부터 운영 제어 체계까지 구축하고, 서비스 적용 가능한 성능과 처리량을 검증한 사례입니다.

ResNet18 FFT FastAPI WandB
상세 이력 보기

문제 상황

현장 환경에서는 엔진 소음, 반사음, 생활 소음 등으로 학습 데이터와 운영 데이터 간 분포 차이가 컸고, 단순 모델 적용만으로는 안정적인 비명 탐지가 어려웠습니다. 실서비스에서는 오탐과 미탐 모두 운영 리스크가 되기 때문에 모델 성능과 운영 정책을 함께 설계해야 했습니다.

해결 방식

  • FFT 기반 특징 추출, Noise Injection, ResNet18 기반 모델링을 적용해 정확도와 추론 비용의 균형을 검증
  • WandB 기반 실험 관리로 하이퍼파라미터 변화에 따른 성능과 지연 시간을 추적하며 운영 가능한 모델을 선택
  • FastAPI 기반 제어 API를 설계해 스트림 상태, 이벤트 발생 여부, 설정 변경을 실시간으로 제어할 수 있는 운영 체계를 구축

성과

총 3,493개 샘플 기준 정확도 95.22%, 특이도 95.97%, 재현율 92.92%를 기록했고, 현실적 활성화 비율 기준 16스트림 환경에서 평균 2초 이내 응답과 최대 9스트림 안정 운영 범위를 검증했습니다. 모델링과 서빙을 함께 설계해 실제 서비스 적용 가능한 AI 운영 구조를 구축했습니다.

Streaming

HLS 기반 대용량 멀티 스트리밍 구조 설계 및 최적화

대용량 CAM 영상을 여러 개 동시에 재생하고 동일 시점으로 제어해야 하는 요구사항을 해결해, 실사용 가능한 멀티 뷰 환경을 구축한 사례입니다.

HLS FFmpeg NGINX Spring MVC
상세 이력 보기

문제 상황

IUU 프로젝트에서는 파일당 약 928MB 수준의 CAM 영상을 한 화면에서 여러 개 동시에 재생하고, 같은 시점으로 제어해야 했습니다. 단일 MP4 방식은 5개 이상의 영상 동시 로드에서 초기 지연과 버퍼링이 심각해 실사용이 어려웠습니다.

해결 방식

  • HLS 방식을 직접 검토하고 제안했으며, EC2, S3, CloudFront, MediaConvert 등 여러 아키텍처를 비용과 운영 측면에서 비교한 뒤 EC2 기반 구조를 선택
  • Codec Copy와 Re-encode 방식을 비교 실험하고, 사용자 경험과 탐색 안정성을 위해 Re-encode 방식을 최종 채택
  • Python Worker 방식의 CPU 부담과 동시 처리 한계를 해결하기 위해 Java 기반 서비스로 리팩토링하고, Semaphore로 동시 인코딩 수를 제어
  • FFmpeg 옵션 튜닝과 키프레임 정렬을 통해 Seek 시 여러 영상이 동일한 시점을 오차 없이 표시하도록 개선

성과

고화질 GB 단위 영상 5개를 동시에 로드하고, 최대 6개 멀티 뷰 동기화 재생 환경을 구축했습니다. 10배속, 10초 전후 이동, 재생/정지 제어를 1초 미만 지연으로 수행할 수 있는 수준까지 최적화해 실사용 가능한 영상 리뷰 환경을 만들었습니다.

MSA · Observability

MSA 기반 스마트 오피스 통합 인프라 및 관측 지표 설계

분산 환경에서 설정 불일치와 장애 추적 어려움을 해결하기 위해 TraceId 기반 로그 모듈과 ELK 관측 체계를 구축한 사례입니다.

Spring Cloud ELK MSA Observability
상세 이력 보기

기술적 페인 포인트

8인의 개발자가 가변적인 서비스들을 개발하면서, 로컬 환경의 설정이 배포 환경에 반영되지 않아 발생하는 런타임 에러가 반복되었습니다. 특히 서비스 간 호출 시 어느 구간에서 지연이나 실패가 발생하는지 알 수 없는 분산 시스템의 비가시성 문제가 시스템 안정성의 최대 걸림돌이었습니다.

엔지니어링 가설 및 설계 방향

  • TraceId를 전파하고 로그 규격을 모듈화해 라이브러리 형태로 강제하면 전체 서비스의 호출 흐름을 단일 타임라인으로 시각화할 수 있을 것이라고 가정
  • 설정 정보를 코드화하여 외부 주입(Config Server) 방식으로 단일화하면 환경 불일치로 인한 장애를 원천 차단하고, 결정론적 배포가 가능하다고 판단

실험 및 기술적 검증 과정

  • TraceId 생성과 response_time 계산을 포함하는 AOP 공통 로그 모듈을 직접 설계하고, JitPack 기반 라이브러리로 배포해 모든 서비스가 동일한 형식의 로그를 남기도록 통일
  • ELK 파이프라인을 구축하고, 컨테이너 간 TZ 동기화와 Logstash 필터 튜닝을 통해 Kibana에서 1ms 단위 요청 흐름 추적이 가능하도록 검증
  • Spring Cloud Config와 GitHub Actions 기반으로 민감 정보와 환경 변수를 중앙 관리하고, 어떤 서비스든 동일한 배포 파이프라인을 타도록 표준화
  • 쾌적도 측정 AI 모델이 수집된 환경 데이터를 실시간으로 처리할 수 있도록 인프라 레이어에 통합하고, AI 연산 부하가 전체 파이프라인 지연에 미치는 영향을 함께 모니터링

결과 및 성과

장애 발생 시 각 서버 로그 파일을 수동 탐색하던 방식을 TraceId 기반 중앙 검색으로 바꾸어 평균 장애 분석 시간(MTTR)을 80% 이상 단축했습니다. 인프라 총괄로서 개별 개발자가 인프라 설정이 아닌 비즈니스 로직에만 집중할 수 있는 공통 인프라 레이어를 제공하는 성과를 거두었습니다.
