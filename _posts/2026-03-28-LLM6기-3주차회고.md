# 모두의 연구소 LLM 6기 재직자 과정 3주차 복습 및 회고

## 3주차

3주차는 지난주처럼 RAG 파이프라인을 더 붙이는 주차라기보다,  
LLM이 더 안정적으로 출력하고, 더 맞게 추론하고, 대화 맥락을 더 잘 유지하도록  
프롬프트와 메모리 구조를 정교하게 설계하는 주차였다.

이번 주의 큰 흐름을 한 줄로 정리하면 아래와 같다.

```text
출력 형식 제어
-> 구조화 추출
-> 제약 프롬프트 설계
-> context 기반 답변
-> zero-shot / few-shot
-> dynamic few-shot + 평가
-> reasoning guardrail
-> chat memory / window / summary memory
```

1주차가 "LLM 호출과 기본 체인",  
2주차가 "검색 기반 답변 시스템"이었다면,  
3주차는 그 위에서 "출력 품질과 대화 품질을 통제하는 방법"을 배우는 흐름에 가까웠다.

---

### 0325 - Structured Prompt, Context, Few-shot

#### 1. 원하는 출력 형식은 애매하게 말하지 말고 구조를 보여줘야 한다

0325 수업에서는 같은 작업도  
자연어로 막연히 요청하는 방식과,  
Markdown이나 JSON 구조를 미리 강하게 제시하는 방식의 차이를 계속 비교했다.

예를 들어 리뷰 분석을 할 때도 그냥 "분석해줘"가 아니라  
필드 이름과 자료형까지 포함한 JSON 예시를 함께 주면 응답이 훨씬 안정적이었다.

```python
json_prompt = """다음 제품 리뷰를 분석해서 json 형식으로 출력하세요

리뷰 : "이 노트북 정말 가벼워서 좋아요! 다만 키보드 타건감이 아쉽네요"
출력형식
{
  "overall_sentiment": "긍정/부정/혼합 중 하나",
  "score": 1~5점 점수,
  "pros": [장점],
  "cons": [단점],
  "keywords": [키워드1, 키워드2, 키워드3]
}
"""
```

이 부분에서 가장 크게 느낀 건,  
모델에게 "잘 정리해줘"라고 말하는 것과  
"이 필드명으로, 이 타입으로, 이 범위만 써"라고 말하는 것은 완전히 다른 수준의 지시라는 점이었다.

#### 2. parser는 만능이 아니다

초반에 특히 중요했던 포인트는  
parser가 형식을 보장해주는 도구가 아니라는 점이었다.

- parser는 잘 나온 출력을 읽고 변환하는 역할에 가깝고
- 모델이 애초에 틀린 형식으로 내보내면 parser도 쉽게 깨질 수 있다

즉 안정적인 구조화 출력은

1. prompt에서 형식을 강하게 지정하고
2. parser로 후처리하고
3. 실패 가능성까지 염두에 두는 방식

으로 접근해야 한다는 걸 배웠다.

#### 3. 반구조화 텍스트는 JSON 스키마로 추출한다

회의 문장을 정리하는 예제도 꽤 인상적이었다.

```python
json_structure = {
    "date": "YYYY-MM-DD",
    "attendees": ["이름1", "이름2"],
    "agenda": ["안건1", "안건2"],
    "decisions": ["결정1"],
    "action_items": [
        {"assignee": "담당자", "task": "작업", "deadline": "기한"}
    ]
}
```

핵심은 자유 텍스트를 그냥 요약하게 하는 것이 아니라  
"이 스키마에 맞춰 추출하라"로 바꾸는 것이다.

이런 방식은 단순 정리를 넘어서

- 후속 자동화에 바로 연결할 수 있고
- 사람이 다시 읽기도 쉽고
- 데이터처럼 다루기 쉬워진다

는 점에서 확실히 실무적이었다.

#### 4. 제약 프롬프트는 길이, 내용, 형식, 스타일로 나누면 설계가 쉬워진다

`ConstrainedPrompt` 예제도 좋았다.

```python
result = (
    ConstrainedPrompt("클라우드 컴퓨팅의 장점을 설명해주세요")
    .add_length("5개의 불릿 포인트")
    .add_content("비용, 확장성, 보안 관점을 반드시 포함")
    .add_format("각 포인트는 한줄로")
    .add_style("IT 비전공 경영진 대상")
    .execute(temperature=0.3)
)
```

그동안 프롬프트를 한 덩어리 문장으로만 쓰는 경우가 많았는데,  
이런 식으로 제약을 축별로 나누니 훨씬 읽기 쉽고 재사용도 쉬웠다.

또 `execute(self, **kwargs)`처럼 실행 옵션을 나중에 유연하게 받는 구조도 같이 배웠는데,  
프롬프트 설계와 호출 옵션을 분리하는 감각이 꽤 중요하게 느껴졌다.

#### 5. context 기반 답변은 "문서 밖으로 못 나가게" 만드는 장치다

회사 정책 문서를 주고  
"반드시 문서 내용만 근거로 답하고, 없으면 없다고 말하라"는 식으로 응답을 제한하는 예제도 나왔다.

이 방식은 결국 RAG의 가장 기본적인 사고방식과 연결된다.

- 모델 일반 지식을 믿지 않고
- 지금 준 문서에만 근거하게 하고
- 추측을 금지한다

즉 이번 주 초반은 새로운 모델 기능을 배우는 것보다  
"모델이 마음대로 답하지 못하게 통제하는 법"을 배우는 느낌이 강했다.

#### 6. zero-shot과 few-shot

0325 후반에는 zero-shot과 few-shot도 같이 다뤘다.

- zero-shot은 카테고리와 규칙만 주고 바로 분류시키는 방식이고
- few-shot은 입력-출력 예시를 먼저 보여줘서 원하는 패턴을 따라오게 만드는 방식이다

특히 비즈니스 톤 변환, 리뷰 감정 분석 같은 작업에서는  
few-shot이 출력 형식과 톤을 훨씬 안정적으로 맞춰준다는 점이 분명했다.

### 0326 - Dynamic Few-shot, Evaluation, Reasoning

#### 1. few-shot도 "어떤 예시를 넣을지"가 중요하다

0326에서는 few-shot을 한 단계 더 확장해서  
현재 입력과 비슷한 예시만 동적으로 고르는 방식을 다뤘다.

```python
example_selector = SemanticSimilarityExampleSelector.from_examples(
    large_examples,
    embeddings,
    InMemoryVectorStore,
    k=3
)
```

이 구조는

- 예시들을 임베딩하고
- 현재 입력과 가까운 예시를 찾고
- 그중 가장 비슷한 몇 개만 prompt에 넣는다

는 흐름으로 동작한다.

즉 fixed few-shot이 항상 같은 예시를 들고 다니는 방식이라면,  
dynamic few-shot은 지금 질문과 가장 비슷한 참고 예시를 골라 붙이는 방식에 가깝다.

#### 2. 문자열 few-shot과 chat few-shot

이날은 `FewShotPromptTemplate`와 `FewShotChatMessagePromptTemplate` 차이도 같이 봤다.

```python
example_prompt_chat = ChatPromptTemplate.from_messages([
    ("human", "리뷰 : {input}"),
    ("ai", "감정 : {output}")
])
```

문자열 기반은 프롬프트를 한눈에 보기 좋고,  
chat 기반은 `system / human / ai` 메시지 구조를 그대로 유지할 수 있어서  
대화형 모델과 더 자연스럽게 연결된다는 점이 흥미로웠다.

#### 3. 프롬프트도 평가가 필요하다

이날 가장 좋았던 부분 중 하나는  
zero-shot과 few-shot을 그냥 느낌으로 비교하지 않았다는 점이었다.

```python
def evaluate_zero_shot(dataset):
    ...

def evaluate_few_shot(dataset, examples):
    ...
```

각 샘플마다

- 원문
- 기대 라벨
- 모델 예측
- 정답 여부

를 남기고 accuracy를 계산하면서  
"좋아 보이는 프롬프트"와 "실제로 더 잘 맞는 프롬프트"는 다를 수 있다는 걸 배웠다.

즉 프롬프트 엔지니어링도 결국 실험과 평가가 필요했다.

#### 4. reasoning guardrail

후반에는 퍼즐형 문제를 이용해서  
LLM이 유명한 문제의 원래 정답을 습관적으로 가져오는 실수를 어떻게 줄일지 다뤘다.

```python
system_prompt = """당신은 퍼즐 전문가입니다. 아래 문제들은 유명한 문제의 변형일 수 있습니다.
원래 문제의 답을 그대로 적용하지 마세요.
문제의 조건을 정확히 읽고 단계별로 생각하세요.
계산 결과가 불가능하다면 불가능하다고 분명하게 답하세요."""
```

특히 몬티홀 변형, 평균 속도 문제 같은 예시를 보면서  
모델은 조건을 조금만 바꿔도 원래 유명한 정답을 그대로 꺼내오려는 경향이 있다는 걸 느꼈다.

그래서

- 원문 문제와 변형 문제를 구분하게 만들고
- 조건을 다시 읽게 만들고
- 단계별 사고를 요구하는 가드레일

이 꽤 중요했다.

이날 내용은 단순히 CoT를 넣는 정도가 아니라,  
"어떤 문제에서 모델이 자주 틀리는가"를 알고 미리 안전장치를 두는 사고방식 자체가 중요했다.

### 0327 - Chat Memory, Window, Summary

#### 1. 모델은 이전 대화를 자동으로 기억하지 않는다

0327은 개인적으로 3주차에서 가장 실무적인 날이었다.

처음 예제부터 아주 단순했다.

```python
llm.invoke("내이름은 인호입니다").content
llm.invoke("내이름은 뭔가요?").content
```

이렇게 따로 호출하면  
두 번째 호출은 첫 번째 내용을 자동으로 기억하지 못한다.

즉 대화형 애플리케이션에서는  
이전 메시지를 직접 다시 함께 보내야 한다는 원리를 먼저 짚고 갔다.

#### 2. 가장 기본은 메시지 리스트를 직접 이어 붙이는 방식이다

```python
messages = [
    SystemMessage(content="당신은 친절한 AI 어시스턴트입니다."),
    HumanMessage(content="내이름은 ABC 입니다. 반가워요"),
]

response1 = llm.invoke(messages)
messages.append(AIMessage(content=response1.content))
messages.append(HumanMessage(content="내이름은 뭔가요?"))
```

이 방식은 직관적이어서 원리를 이해하기엔 좋지만,

- 메시지 관리가 번거롭고
- 세션 분리가 어렵고
- 대화가 길어질수록 토큰이 계속 커진다

는 단점이 분명했다.

#### 3. 대화 길이는 곧 비용 문제다

중간에는 `tiktoken`으로 메시지 토큰 수를 직접 세어보기도 했다.

```python
def count_tokens(messages):
    total = 0
    for msg in messages:
        total += len(enc.encode(msg.content))
        total += 4
    return total
```

여기서 다시 느낀 건  
memory 설계는 단순히 "얼마나 많이 기억할까"의 문제가 아니라  
"비용, 속도, 품질을 어떻게 균형 잡을까"의 문제라는 점이었다.

#### 4. Buffer / InMemoryChatMessageHistory / RunnableWithMessageHistory

이날은 메모리 관련 도구들이 단계적으로 나왔다.

`ConversationBufferMemory`는 전체 대화를 그대로 쌓는 가장 단순한 방식이었고,

```python
memory = ConversationBufferMemory(return_messages=True)
memory.save_context({"input": "안녕하세요 저는 abc 입니다."}, {"output": "안녕하세요 abc님!"})
```

`InMemoryChatMessageHistory`는 더 저수준의 히스토리 저장소 느낌이었다.

```python
chat_history = InMemoryChatMessageHistory()
chat_history.add_user_message("안녕하세요! 저는 abc입니다.")
chat_history.add_ai_message("저는 누구인가요?")
```

가장 실전적인 건 `RunnableWithMessageHistory`였다.

```python
chain_with_history = RunnableWithMessageHistory(
    chain,
    get_session_history,
    input_messages_key="input",
    history_messages_key="history",
)
```

이걸 쓰면

- 세션별 히스토리를 분리할 수 있고
- 체인에 history를 자동으로 연결할 수 있고
- 직접 메시지를 하나씩 이어붙이는 수고를 줄일 수 있다

즉 "사용자별 대화 세션"을 어떻게 다룰지 처음으로 코드 수준에서 명확하게 본 느낌이었다.

#### 5. window memory

대화가 길어질수록 모든 히스토리를 다 보내는 건 비효율적이기 때문에  
최근 몇 턴만 유지하는 window memory도 실습했다.

```python
window_memory = ConversationBufferWindowMemory(k=2, return_messages=True)
```

`k=2`면 최근 두 개의 대화 턴만 남긴다는 뜻이다.

이 방식은

- 최근 문맥이 더 중요한 챗봇
- 토큰 비용을 관리해야 하는 서비스

에 잘 맞는다는 점이 분명했다.

#### 6. trim_messages와 버전 차이

`trim_messages`를 직접 쓰는 예제도 있었는데,  
여기서 라이브러리 버전별 인자 차이가 꽤 중요했다.

```python
trimmer = trim_messages(
    max_tokens=4,
    strategy="last",
    token_counter=len,
    start_on="human",
)
```

예전 예제처럼 `tokenizer=`를 쓰면 현재 버전에서는 에러가 나고,  
지금 환경에서는 `token_counter=`를 써야 했다.

작은 차이 같지만 실제 실습에서는 이런 버전별 시그니처 차이가 꽤 자주 문제를 만든다는 걸 다시 느꼈다.

#### 7. 직접 슬라이딩 윈도우를 구현할 때의 주의점

리스트를 직접 잘라 최근 K턴만 남기는 구현도 했다.

```python
def add_turn(user_input, ai_output):
    messages.append(HumanMessage(content=user_input))
    messages.append(AIMessage(content=ai_output))
    del messages[:-(K * 2)]
```

여기서 중요한 건  
`messages = messages[-(K*2):]`처럼 함수 안에서 재할당하면  
지역 변수 스코프 문제를 만들 수 있다는 점이었다.

즉 메모리 로직은 단순한 것 같아도  
Python 리스트 수정 방식이나 스코프 규칙까지 같이 알아야 안정적으로 짤 수 있었다.

#### 8. summary memory

오래된 대화를 통째로 버리기보다 요약해서 압축하는 summary memory도 나왔다.

```python
class SummaryMemory:
    def __init__(self, summary_interval=3):
        self.summary = ""
        self.recent_messages = []
        self.turn_count = 0
        self.summary_interval = summary_interval
```

이 구조는

- 최근 대화는 그대로 보관하고
- 일정 턴이 지나면 오래된 대화를 요약으로 압축하는 방식이다

window memory보다 구현은 조금 더 복잡하지만,  
긴 대화에서 핵심 맥락만 유지하고 싶을 때 더 실용적일 것 같았다.

## 회고

3주차는 지난주보다 훨씬 덜 화려해 보일 수도 있지만,  
개인적으로는 LLM 애플리케이션을 실제로 더 안정적으로 만드는 데 필요한 포인트가 가장 많이 들어 있던 주차였다.

특히 크게 남은 건 아래 네 가지였다.

1. 출력 품질은 모델 성능보다 prompt 구조화가 먼저다.
2. few-shot은 단순 예시 나열이 아니라 형식과 판단 기준을 심는 작업이다.
3. 프롬프트도 감으로 고르는 게 아니라 평가 함수와 지표로 비교해야 한다.
4. 대화형 앱에서는 memory 전략이 곧 비용과 품질 설계다.

이번 주를 지나면서  
LLM을 잘 쓰는 방법이 단순히 좋은 질문을 던지는 것이 아니라,

- 어떤 형식으로 답하게 할지
- 어떤 예시를 붙일지
- 어떻게 평가할지
- 어떤 맥락을 남기고 버릴지

까지 포함하는 훨씬 더 구조적인 문제라는 걸 확실히 느꼈다.
