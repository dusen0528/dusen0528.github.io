# 모두의 연구소 LLM 6기 재직자 과정 5주차 1일차 복습 및 정리

## 5주차 1일차

5주차 1일차는 retrieval 단계에서 한 번 찾은 후보 문서를  
그대로 생성에 넘기지 않고,  
한 번 더 다시 정렬하는 `reranking`을 중심으로 정리한 날이었다.

이번에는 Transformer와 BERT의 관계를 다시 짚고,  
Bi-Encoder 기반 벡터 검색이 왜 빠른지 확인한 뒤,  
BM25, Cross-Encoder, LLM reranking이 각각 어떤 역할을 하는지 비교했다.

이번 수업의 큰 흐름을 한 줄로 정리하면 아래와 같다.

```text
Transformer와 BERT 관계 정리
-> Bi-Encoder 기반 retrieval 복습
-> BM25와 sparse signal 이해
-> Cross-Encoder / LLM reranking 비교
-> keyword rerank 구현
-> rank change 추적
-> LLM rerank JSON 파싱
-> BM25 + LLM hybrid rerank
-> retrieval -> reranking -> generate 흐름 정리
```

이번 수업에서 가장 크게 느낀 점은,  
retrieval은 후보를 빨리 모으는 단계이고  
reranking은 그 후보 중에서 정말 더 관련 있는 문서를 앞에 세우는 단계라는 점이었다.

즉 검색을 한 번으로 끝내지 않고  
2단계로 나눠 생각하는 관점이 더 선명해졌다.

---

### Transformer, BERT, Bi-Encoder

#### 1. Transformer와 BERT의 관계를 다시 정리했다

이번 시간에는 먼저  
`Transformer`와 `BERT`의 관계를 다시 정리했다.

- Transformer는 모델 구조이고
- BERT는 Transformer의 encoder를 기반으로 만든 구체적인 모델이다

이걸 다시 보면서 느낀 건,  
이름을 자주 듣다 보면 둘이 같은 층위의 개념처럼 헷갈릴 수 있는데  
사실은 구조와 그 구조를 활용한 모델의 관계라는 점이었다.

기본 개념이지만 retrieval과 reranking 얘기를 할 때  
Bi-Encoder, Cross-Encoder가 모두 Transformer 계열 모델 구조와 연결되기 때문에  
이 정리를 먼저 해두는 게 꽤 중요하게 느껴졌다.

#### 2. 벡터 검색은 Bi-Encoder라서 빠르다

벡터 검색은 query와 document를 각각 따로 임베딩한 뒤  
두 벡터의 유사도를 계산하는 방식이다.

```text
query -> embedding -> query_emb
doc -> embedding -> doc_emb
similarity(query_emb, doc_emb) -> top-k 문서
```

이 방식이 retrieval에 적합한 이유는 명확했다.

- 문서 임베딩을 미리 만들어 둘 수 있고
- 검색 시에는 query 임베딩만 새로 만들면 되고
- 이후엔 빠르게 벡터 유사도 계산만 하면 되기 때문이다

즉 Bi-Encoder 기반 retrieval은  
"많은 문서 중 후보를 빠르게 고르는 1차 필터" 역할에 잘 맞는다.

---

### BM25와 Reranking

#### 1. BM25는 여전히 강한 키워드 신호다

이번 수업에서는 BM25도 다시 나왔다.

BM25는 의미 유사도보다  
query 단어가 문서에 실제로 등장하는지를 강하게 본다.

특히 아래 같은 특성이 다시 정리되었다.

- 너무 흔한 단어는 중요도를 낮추고
- 문서가 너무 길면 길이 보정을 하고
- 정확한 키워드 매칭에 강하다

이걸 보면서 느낀 건,  
vector retrieval이 좋아졌다고 해도  
BM25가 완전히 사라지는 건 아니라는 점이었다.

고유명사, 상품명, 특정 용어 검색에서는  
여전히 sparse signal이 꽤 강하게 작동할 수 있다.

#### 2. retrieval 이후에 다시 순서를 바꾸는 이유

이번 수업의 핵심은 `reranking`이었다.

retrieval 단계에서는 일단 top-k 후보를 빠르게 모으고,  
reranking 단계에서는 그 후보들을 query와 더 정밀하게 다시 비교해서  
순서를 재배치한다.

즉 전체 문서를 처음부터 정밀 비교하는 건 너무 비싸니까

1. 빠른 검색으로 후보를 모으고
2. 좁혀진 후보에만 더 비싼 비교를 한다

는 2단계 구조로 가는 것이다.

이 흐름이 굉장히 현실적으로 느껴졌다.

실제 서비스에서는 속도도 중요하고 품질도 중요하니까,  
retrieval과 reranking을 나누는 게 자연스러운 설계처럼 보였다.

---

### Cross-Encoder와 LLM Reranking

#### 1. Cross-Encoder는 query와 문서를 같이 본다

`Bi-Encoder`는 query와 문서를 따로 임베딩하지만,  
`Cross-Encoder`는 query와 document를 한 쌍으로 같이 넣고  
관련성 점수를 직접 계산한다.

```text
(query, doc) -> relevance score
```

이 방식의 장점은 더 정밀하다는 점이다.

단어 하나하나와 문맥 관계를 같이 보면서  
정말 관련 있는 문서인지 더 세밀하게 판단할 수 있다.

대신 비용은 더 크다.

모든 문서를 이렇게 비교할 수는 없으니  
결국 retrieval 뒤의 reranking 단계에 어울린다는 점도 자연스럽게 이해되었다.

#### 2. LLM reranking은 더 유연하지만 파싱이 중요하다

이번 실습에서는 LLM에게 후보 문서들을 보여주고,  
문서 id별 관련성 점수를 JSON으로 받는 방식도 다뤘다.

예를 들면 아래 같은 형태다.

```json
{"scores": {"d1": 0.91, "d2": 0.97, "d3": 0.22}}
```

이 방식의 장점은 분명했다.

- query와 문서를 함께 읽고
- 의미적으로 더 유연하게 판단할 수 있고
- 설명 가능한 근거를 붙일 수도 있다

하지만 동시에 실무적인 주의점도 컸다.

- 빈 문자열이면 `json.loads()`가 실패하고
- 설명 문장이 섞여도 실패하고
- 코드블록이 붙어도 실패할 수 있다

즉 LLM reranking은  
모델 성능뿐 아니라  
"응답 형식을 얼마나 안정적으로 강제하고 파싱하느냐"가 아주 중요했다.

---

### 코드 구현에서 놓치기 쉬운 포인트

#### 1. `(doc, score)`를 `doc`처럼 다루면 바로 꼬인다

이번 실습에서 꽤 현실적이었던 부분은  
검색 결과가 보통 `Document` 하나가 아니라  
`(doc, score)` 튜플 리스트라는 점을 계속 확인한 것이었다.

잘못 쓰면 아래처럼 된다.

```python
for doc in search_results:
    print(doc.metadata["id"])
```

하지만 실제로는 unpacking을 해야 한다.

```python
for doc, score in search_results:
    print(doc.metadata["id"], score)
```

아주 작은 차이처럼 보이지만,  
retrieval 파이프라인에서는 이런 자료형 이해가 꼬이면  
후반부 rank 비교나 reranking 로직 전체가 흔들릴 수 있다는 걸 다시 느꼈다.

#### 2. rank change는 `doc id` 기준으로 보는 게 안전하다

reranking 전후 변화를 비교할 때  
점수 자체로 문서를 찾으면 잘 안 맞을 수 있다.

reranking 후에는 점수가 바뀌기 때문이다.

그래서 원래 순위를 저장할 때도

```python
orig_ranks = {
    doc.metadata["id"]: i + 1
    for i, (doc, _) in enumerate(original_results)
}
```

처럼 `doc id` 기준으로 맵을 만드는 게 더 안전했다.

이걸 보면서 다시 느낀 건,  
검색 실험에서는 문서 식별자가 정말 중요하다는 점이었다.

점수는 바뀌어도 id는 남기 때문에  
평가와 디버깅의 기준점 역할을 해준다.

---

### RAG 전체 흐름

#### 1. retrieval과 reranking을 나누면 파이프라인이 더 선명해진다

이번 수업의 전체 흐름은 결국 아래로 정리되었다.

1. 사용자 질문 입력
2. 벡터 검색 또는 BM25로 top-k 후보 문서 검색
3. 후보 문서에 대해 reranking 적용
4. reranked된 상위 문서를 선택
5. 선택된 문서를 context로 넣어 LLM이 답변 생성

즉

```text
retrieval -> top-k 후보 확보 -> reranking -> 상위 n개 선택 -> generate
```

이 구조를 보면서  
이제 RAG를 훨씬 레이어별로 생각하게 되었다.

- retrieval은 후보 회수 문제
- reranking은 후보 정렬 문제
- generation은 선택된 근거를 바탕으로 답하는 문제

즉 세 단계를 나눠 보면  
어디를 개선해야 하는지도 훨씬 분명해진다.

#### 2. hybrid rerank도 충분히 실전적이다

후반부에는 BM25 점수와 LLM 점수를 각각 구한 뒤  
정규화해서 가중합하는 hybrid rerank도 다뤘다.

예를 들어

- `0.3 * BM25 + 0.7 * LLM`

같은 방식으로 최종 점수를 만들 수 있다.

이 방식이 인상 깊었던 이유는  
키워드 일치 신호와 의미 기반 판단을  
둘 다 반영할 수 있기 때문이다.

즉 retrieval 단계뿐 아니라 reranking 단계에서도  
hybrid 사고방식이 충분히 유효하다는 걸 느꼈다.

---

## 정리

5주차 1일차는 retrieval 다음 단계인 reranking을 본격적으로 정리한 날이었다.

특히 아래 내용이 핵심으로 남았다.

1. Bi-Encoder 기반 retrieval은 빠르게 후보를 모으는 데 적합하다.
2. Cross-Encoder와 LLM reranking은 더 정밀하게 관련성을 다시 판단하는 데 적합하다.
3. BM25는 retrieval뿐 아니라 reranking 신호로도 여전히 유효하다.
4. 검색 결과를 다룰 때는 `(doc, score)` 구조와 `doc id` 기준 추적을 정확히 이해해야 한다.
5. RAG는 retrieval, reranking, generation을 분리해서 보면 훨씬 설계가 선명해진다.

개인적으로는 이번 수업을 통해  
RAG 품질을 높이는 일이 단순히 검색 잘하기에서 끝나는 게 아니라  
찾아온 후보를 다시 얼마나 잘 정렬하느냐까지 포함한다는 점이 더 또렷해졌다.

앞으로는 top-k만 높이는 데 집중하기보다,  
retrieval로 후보를 넉넉히 모은 뒤  
reranking으로 정말 필요한 문서를 앞에 세우는 구조를 더 자주 떠올리게 될 것 같다.
