# 모두의 연구소 LLM 6기 재직자 과정 4주차 4일차 복습 및 정리

## 4주차 4일차

4주차 4일차는 검색기를 만드는 데서 더 나아가,  
"질문을 어떻게 바꾸고", "여러 결과를 어떻게 합치고",  
"어떤 설정이 실제로 더 좋은지"까지 튜닝하는 시간이었다.

전날까지는 keyword search, vector search, BM25, hybrid search를 비교하는 흐름이었다면,  
이번에는 그 위에 query expansion, RRF, multi-query, HyDE 같은 전략을 붙여서  
retrieval 품질을 한 번 더 끌어올리는 쪽으로 확장되었다.

이번 수업의 큰 흐름을 한 줄로 정리하면 아래와 같다.

```text
FAISS 검색 함수 구성
-> Query Expansion
-> 확장 결과 병합
-> RRF로 순위 기반 결합
-> Multi-Query 생성
-> query diversity 확인
-> HyDE 검색
-> Precision / Recall / MRR 평가
-> Grid Search로 하이브리드 가중치 튜닝
```

이번 수업에서 가장 크게 느낀 점은,  
검색 품질을 높인다는 건 검색 모델 하나를 바꾸는 문제가 아니라  
질문 자체를 더 잘 검색되게 바꾸는 문제이기도 하다는 점이었다.

---

### Query Expansion

#### 1. 원래 질문 하나만으로는 검색 단서가 부족할 수 있다

이번 수업에서는 원래 질문을 더 잘 검색될 수 있는 표현으로 넓히는  
`Query Expansion`을 먼저 다뤘다.

예를 들어 질문이 아래처럼 들어오면

- `AI 챗봇 답변 품질 높이기`

이걸 그대로 한 번만 검색하는 대신,

- `RAG 검색 증강 생성 기법`
- `프롬프트 엔지니어링 LLM 입력 설계`

처럼 관련 표현을 함께 던지는 방식이다.

이걸 보면서 느낀 건,  
사용자 질문은 짧고 모호한 경우가 많아서  
검색기 입장에서는 단서가 충분하지 않을 수 있다는 점이었다.

즉 Query Expansion은  
"질문을 더 잘 찾히는 말로 보강하는 전략"이라고 이해하면 가장 편했다.

#### 2. 확장 후에는 결과를 어떻게 합칠지도 중요하다

노트북에서는 원래 질문과 확장 질문들을 모두 검색한 뒤,  
문서별 최고 점수만 남기는 방식으로 결과를 합쳤다.

```python
for q in [original] + expanded_list:
    for idx, score in search_faiss(q, top_k=top_k):
        if idx not in all_results or score > all_results[idx]:
            all_results[idx] = score
```

이 방식의 장점은 꽤 단순명확했다.

- 구현이 쉽고
- 여러 질의 결과를 빠르게 병합할 수 있고
- 확장 쿼리 덕분에 새 문서를 더 회수할 수 있다

즉 확장만 하는 것으로 끝나는 게 아니라,  
그 결과를 어떤 기준으로 모아서 top-k를 다시 만들지까지 같이 생각해야 했다.

---

### 확장 전후 비교

#### 1. Query Expansion 효과는 새로 들어온 문서를 보면 감이 온다

이번 실습에서는 확장 전 검색 결과와 확장 후 검색 결과를  
문서 id 집합으로 비교해서  
새롭게 들어온 문서를 확인했다.

```python
without_ids = set(idx for idx, _ in without)
with_exp_ids = set(idx for idx, _ in with_exp)
new_doc = with_exp_ids - without_ids
```

이 비교가 좋았던 이유는  
단순히 점수만 보는 것보다 효과가 훨씬 직관적으로 보였기 때문이다.

- 원래 질문만 썼을 때는 안 잡히던 문서가
- 확장 쿼리를 넣은 뒤에는 들어온다

이 흐름을 눈으로 확인하니까,  
Query Expansion이 왜 recall 쪽에서 자주 쓰이는지 더 잘 이해되었다.

#### 2. 확장은 좋지만 무조건 좋은 건 아니다

다만 여기서 같이 느낀 건,  
확장을 잘못하면 관련 없는 문서까지 더 많이 끌고 올 수도 있다는 점이었다.

즉 query expansion은 recall을 올리기 좋지만,  
그만큼 precision이 흔들릴 가능성도 있다.

그래서 결국은  
"확장했더니 뭘 더 찾았는가"와  
"그게 정말 도움이 되는 문서인가"를 같이 봐야 한다는 생각이 들었다.

---

### RRF

#### 1. 여러 결과를 합칠 때 점수 대신 순위를 쓰는 방식

이번 수업에서 꽤 인상 깊었던 개념이 `RRF(Reciprocal Rank Fusion)`였다.

```python
score += 1 / (k + rank)
```

핵심 아이디어는 단순하다.

- 각 검색 결과에서 상위에 나온 문서일수록 높은 점수를 받고
- 여러 결과 목록에서 반복해서 상위권에 등장하면 더 유리해진다

이 방식이 좋은 이유는  
검색 결과마다 점수 스케일이 달라도 잘 동작하기 때문이다.

어떤 검색기는 cosine similarity를 쓰고,  
어떤 검색기는 BM25 점수를 쓸 수 있는데  
이 둘의 절대값은 직접 비교하기 어렵다.

하지만 순위는 비교할 수 있다.

즉 RRF는  
"서로 다른 검색 결과를 순위 기준으로 안정적으로 합치는 방법"이라고 이해하게 되었다.

#### 2. Query Expansion과 RRF는 같이 쓸 때 더 힘이 난다

이번 실습에서는 여러 확장 쿼리를 만든 뒤  
각 결과를 RRF로 다시 합쳤다.

```python
for query in queries:
    results = search_faiss(query, top_k=5)
    for rank, (doc_id, _) in enumerate(results, 1):
        rrf_scores[doc_id] += 1 / (k + rank)
```

이 구조를 보면서 확실히 정리된 건 아래였다.

- Query Expansion은 검색 입력을 넓히는 역할
- RRF는 그 여러 입력의 결과를 잘 합치는 역할

즉 둘은 경쟁 관계가 아니라  
앞단과 뒷단에서 서로 보완하는 조합에 가깝다.

---

### Multi-Query와 Query Diversity

#### 1. 사람이 일일이 확장하지 않고 LLM이 다시 써준다

후반부에는 `Multi-Query`도 다뤘다.

이건 사람이 직접 확장 쿼리를 쓰는 대신,  
LLM에게 원래 질문을 여러 관점으로 다시 써 달라고 요청하는 방식이다.

예를 들어

- `벡터검색`

이라는 짧은 질문을 아래처럼 바꿀 수 있다.

- `벡터 기반 검색`
- `벡터를 활용한 정보 검색`
- `벡터를 이용한 검색 방법`

즉 핵심은  
"사람 손으로 확장 쿼리를 만들지 않고 LLM이 여러 버전을 생성하게 하는 것"이다.

#### 2. 출력 파싱이 생각보다 중요했다

실습에서는 LLM이 결과를 깔끔하게 줄 것 같지만  
실제로는 `재작성1:` 같은 라벨을 붙이거나  
빈 줄이 섞여 나오는 경우가 있었다.

그래서

- 빈 줄 제거
- `:` 뒤 실제 질의만 추출
- 검색에 바로 쓸 수 있는 리스트로 정리

같은 후처리가 필요했다.

이 부분이 좋았던 건,  
LLM을 붙인 기능은 프롬프트만 잘 쓰면 끝나는 게 아니라  
출력을 안정적으로 소비하는 단계까지 구현해야 한다는 점을 다시 느끼게 해줬다는 것이다.

#### 3. query diversity도 같이 봐야 한다

노트북에는 생성된 쿼리들이 서로 얼마나 다른지 보는  
`query_diversity`도 있었다.

```python
diversity = 1 - avg_cosine_similarity
```

이 지표를 보면서 느낀 건,  
확장 쿼리가 많다고 무조건 좋은 건 아니라는 점이었다.

서로 거의 같은 표현만 세 개 만들어오면  
실제로는 검색 관점을 넓힌 게 아니기 때문이다.

즉 Multi-Query에서는  
"몇 개를 만들었는가"보다  
"정말 다른 관점이 들어갔는가"가 더 중요했다.

---

### HyDE와 Retrieval 튜닝

#### 1. 질문 대신 가상의 답변 문서를 만들어 검색한다

이번에 새로 배운 전략 중 가장 흥미로웠던 건 `HyDE`였다.

흐름은 아래처럼 정리할 수 있다.

1. 질문을 받는다
2. LLM이 그 질문에 대한 가상의 짧은 답변 문서를 만든다
3. 그 문서를 임베딩한다
4. 그 임베딩으로 벡터 검색을 한다

즉 질문을 직접 검색하는 대신,  
"이 질문에 대한 답변이라면 이런 문서일 것 같다"는 텍스트를 먼저 만들어서  
그걸 검색에 쓰는 방식이다.

질문이 너무 짧거나 애매할 때는  
이 가상 문서가 검색 단서를 더 풍부하게 만들어 줄 수 있다는 점이 인상 깊었다.

#### 2. 결국 비교는 metric으로 해야 한다

수업 후반에는 precision, recall, MRR로 전략별 결과를 비교하고,  
BM25와 vector를 섞는 하이브리드 검색에서는  
`w_bm25` 값을 grid search로 바꿔가며 최적 가중치를 찾았다.

```python
for w in weights:
    retrieved = hybrid_search(item["query"], w_bm25=w, top_k=5)
```

이 부분이 특히 좋았던 이유는  
검색 전략을 감으로 선택하지 않고  
평가 지표로 비교하는 습관을 계속 이어간다는 점이었다.

예전에는 query expansion이나 multi-query 같은 게  
"뭔가 좋아 보이는 기법" 정도로 느껴졌는데,  
이제는 실제로 precision, recall, MRR이 어떻게 변하는지로 봐야 한다는 생각이 훨씬 강해졌다.

## 정리

4주차 4일차는 retrieval을 더 잘 튜닝하는 전략들을 한 번에 정리한 날이었다.

특히 아래 내용이 핵심으로 남았다.

1. Query Expansion은 질문의 검색 단서를 보강해서 더 많은 관련 문서를 회수하게 도와준다.
2. RRF는 서로 다른 검색 결과를 점수 대신 순위 기준으로 안정적으로 합칠 수 있게 해준다.
3. Multi-Query와 HyDE는 질문 자체를 더 검색 친화적인 형태로 바꾸는 전략이다.
4. 어떤 전략이 정말 좋은지는 precision, recall, MRR 같은 지표로 비교해야 한다.

개인적으로는 이번 수업을 통해  
retrieval 개선이 단순히 임베딩 모델 교체나 top-k 조정 수준이 아니라,  
질문 생성, 결과 결합, 파라미터 탐색까지 이어지는 꽤 넓은 실험 문제라는 걸 더 체감했다.

앞으로 RAG를 만질 때도  
원래 질문을 그대로 던지는 것만 고집하기보다,  
query expansion이나 multi-query처럼  
질문을 더 잘 검색되게 바꾸는 관점도 같이 챙겨봐야 할 것 같다.
