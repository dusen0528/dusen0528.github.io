# 모두의 연구소 LLM 6기 재직자 과정 4주차 5일차 복습 및 정리

## 4주차 5일차

4주차 5일차는 주중에 배운 retrieval 평가와 hybrid search 개념을  
실제 미니 프로젝트 형태로 묶어보는 시간에 가까웠다.

이번에는 ETF 금융 상품 추천 시스템이라는 도메인을 잡고,  
문서셋을 만들고, 평가 질의를 만들고,  
FAISS와 BM25를 비교하고,  
하이브리드 검색 가중치를 튜닝하고,  
마지막에는 검색 비교 대시보드까지 붙였다.

이번 흐름을 한 줄로 정리하면 아래와 같다.

```text
ETF 데이터셋 준비
-> 사용자 페르소나와 질의 정의
-> 추가 ETF 문서 생성
-> 평가용 multi-relevance query 구성
-> FAISS / MMR / BM25 / Hybrid 검색 비교
-> Hit Rate / MRR / NDCG 리포트 작성
-> alpha grid search
-> 동의어 확장 / custom multi-query
-> Gradio 비교 대시보드 구성
```

이번 실습에서 가장 크게 느낀 점은,  
RAG 프로젝트는 "검색이 되느냐"만 보는 게 아니라  
도메인 데이터, 사용자 질문, 평가셋, UI까지 다 연결해서 봐야  
조금 더 실제 서비스처럼 보인다는 점이었다.

---

### ETF 데이터셋과 사용자 페르소나

#### 1. 검색 프로젝트는 도메인 데이터부터 시작한다

이번 실습에서는 ETF 금융 상품 데이터를 기반으로  
검색형 RAG 시스템을 구성했다.

문서 하나하나는 ETF 이름, 카테고리, 설명, 수수료, AUM, 수익률, 키워드 같은 정보를 가지고 있었다.

이 구성을 보면서 느낀 건,  
실제 추천형 검색에서는 단순 텍스트 몇 줄보다  
구조화된 메타데이터가 같이 있을 때 훨씬 해석하기 쉽다는 점이었다.

예를 들어 검색 결과를 보고도

- 어떤 카테고리인지
- 위험도가 어떤지
- 어떤 투자자에게 맞는지

를 바로 연결해서 볼 수 있다.

즉 retrieval 문서셋도  
"문장 저장소"라기보다  
"추천 판단에 필요한 정보 묶음"으로 준비해야 한다는 느낌이 더 강했다.

#### 2. 사용자 페르소나가 질의 설계 기준이 된다

이번에는 `초보`, `중급`, `전문` 투자자 페르소나를 나누고,  
각 사용자 유형마다 여러 검색 질의를 정의했다.

예를 들면 아래와 같은 식이었다.

- 초보: `안전한 투자 상품 추천해주세요`
- 중급: `S&P500 추종 ETF 비교`
- 전문: `AI 반도체 관련 ETF 섹터 분석`

이 부분이 좋았던 이유는  
평가 질의가 그냥 랜덤 질문 모음이 아니라  
실제 사용자 관점에서 만들어졌다는 점이었다.

즉 검색 품질을 평가할 때도  
"누가 이런 질문을 할까"를 먼저 생각해야  
질문셋이 더 현실적으로 만들어진다는 걸 느꼈다.

---

### 문서 확장과 평가용 질의셋

#### 1. 추가 ETF 문서를 LLM으로 생성하는 흐름

이번 실습에서는 기존 샘플 ETF 외에도  
ESG, 2차전지, 헬스케어, 리츠, 원자재 같은 추가 ETF 문서를 더 만들었다.

이때 LLM에게 아래 항목을 포함해 설명을 생성하게 했다.

1. 투자 전략
2. 주요 편입 종목
3. 수수료 및 비용
4. 적합한 투자자 유형
5. 주의사항

이 과정을 보면서 느낀 건,  
LLM은 단순 답변 생성뿐 아니라  
검색용 코퍼스를 보강하는 데도 꽤 유용하다는 점이었다.

물론 실제 서비스라면 팩트 검증이 반드시 필요하겠지만,  
실습 단계에서 도메인 문서를 빠르게 확장하는 데는 확실히 편했다.

#### 2. 평가 질의는 multi-relevance 구조가 중요했다

이번 프로젝트에서는 어떤 질의가 정답 문서 하나만 가지는 게 아니라,  
관련 문서가 여러 개일 수 있는 `multi-relevance` 구조도 사용했다.

예를 들어

- `미국 시장에 투자할 수 있는 ETF 비교`

같은 질문은 하나의 ETF만 정답이 아니라  
S&P500, 나스닥, 미국 배당 ETF처럼  
여러 문서가 모두 관련될 수 있다.

이 구조를 보면서 확실히 느낀 건,  
실제 검색에서는 정답이 하나가 아닌 경우가 훨씬 자연스럽다는 점이었다.

그래서 evaluation도  
단순 exact match보다는  
여러 관련 문서를 얼마나 잘 상위에 배치하는가를 봐야 한다는 생각이 더 강해졌다.

---

### MMR, Hit Rate, MRR, NDCG

#### 1. similarity search와 MMR는 목적이 조금 다르다

이번 실습에서는 기본 similarity search와 함께  
`MMR(Max Marginal Relevance)` 검색도 비교했다.

MMR은 단순히 query와 가장 비슷한 문서만 고르는 게 아니라,  
이미 뽑힌 문서들과 너무 비슷한 결과는 줄이고  
조금 더 다양한 결과를 상위에 섞으려는 방식이다.

즉

- similarity search는 관련성 중심
- MMR search는 관련성 + 다양성 균형

이라고 보면 이해하기 쉬웠다.

같은 ETF 카테고리만 몰려 나오는 문제를 완화할 수 있다는 점에서  
추천형 검색과도 잘 어울리는 방식처럼 느껴졌다.

#### 2. Hit Rate, MRR, NDCG를 함께 보는 이유

이번 프로젝트에서는 `Hit Rate`, `MRR`, `NDCG`까지 같이 계산했다.

각 지표의 느낌은 아래처럼 정리할 수 있었다.

- `Hit Rate`: top-k 안에 정답이 하나라도 들어왔는가
- `MRR`: 첫 정답이 얼마나 빨리 등장했는가
- `NDCG`: 관련성이 높은 문서가 상위에 잘 정렬되었는가

특히 NDCG가 들어오면서  
단순히 정답 유무를 넘어서  
"더 중요한 문서를 더 위에 올렸는가"까지 보게 된 점이 인상 깊었다.

즉 retrieval 평가도 점점  
있다/없다 수준에서  
순위 품질까지 보는 방향으로 깊어진 느낌이었다.

---

### BM25 vs FAISS vs Hybrid

#### 1. BM25와 FAISS는 잘하는 문제가 다르다

중간 실습에서는 BM25와 FAISS를 실제로 비교했다.

여기서 다시 확인된 건  
keyword 기반 검색과 vector 기반 검색은  
강점이 다르다는 점이었다.

- BM25는 정확한 키워드 일치에 강하고
- FAISS는 의미적으로 비슷한 표현을 찾는 데 더 유연하다

ETF 검색 같은 도메인에서는  
고유명사, 시장명, 카테고리명처럼 정확한 용어도 중요하고,  
질문 표현 차이도 꽤 크다.

그래서 한쪽만 쓰는 것보다  
둘을 같이 보는 쪽이 더 현실적으로 느껴졌다.

#### 2. alpha 튜닝은 감이 아니라 실험이다

후반부에서는 하이브리드 검색에서  
BM25와 FAISS를 어떤 비율로 섞을지를 `alpha`로 조정했다.

```python
hybrid_scores[ticker] = alpha * faiss_score + (1 - alpha) * bm25_score
```

그리고 `alpha=0.0`부터 `1.0`까지 바꿔 가며  
Hit Rate@5를 비교해서 최적값을 찾았다.

이 흐름이 좋았던 이유는  
하이브리드 검색도 결국 "적당히 0.5"가 아니라  
데이터셋에 맞는 값을 실험으로 찾아야 한다는 걸 보여줬기 때문이다.

즉 hybrid retrieval은  
개념적으로 멋져 보이는 것보다  
평가셋 기준으로 실제 성능이 좋아지는지가 더 중요했다.

---

### 동의어 확장과 Custom Multi-Query

#### 1. 도메인 특화 동의어 사전이 생각보다 중요하다

이번 프로젝트에서는 ETF, 배당, 안정, 성장, 미국 같은 키워드에 대해  
동의어 사전을 만들고 query expansion도 직접 적용했다.

예를 들면

- `ETF` -> `상장지수펀드`, `인덱스펀드`
- `배당` -> `월배당`, `고배당`, `인컴`
- `미국` -> `해외`, `미국주식`, `S&P500`, `나스닥`

이 과정을 보면서  
도메인 검색에서는 일반적인 확장보다  
분야 특화 어휘 사전이 훨씬 직접적으로 먹히는 경우가 많겠다는 생각이 들었다.

특히 금융 쪽은 같은 의미를 다른 투자 용어로 표현하는 경우가 많아서  
동의어 확장이 recall 개선에 꽤 유용해 보였다.

#### 2. Multi-Query도 도메인 프롬프트가 붙으면 더 현실적이다

이번에는 `ETF 검색 전문가` 역할을 주는 custom prompt로  
Multi-Query Retriever도 구성했다.

여기서 좋았던 점은  
그냥 질문을 세 개로 바꾸라고 하는 것보다  
도메인 역할을 명확히 주면 검색에 필요한 표현이 더 자연스럽게 나온다는 점이었다.

즉 multi-query도  
범용 프롬프트 하나로 끝내기보다  
도메인 맥락을 붙였을 때 더 실전형으로 바뀐다는 걸 느꼈다.

---

### 검색 비교 대시보드

#### 1. 마지막에 UI를 붙이니 전체 흐름이 더 잘 보였다

실습 마지막에는 Gradio로  
FAISS, BM25, Hybrid 결과를 나란히 보여주는 비교 UI도 만들었다.

이 부분이 꽤 좋았던 이유는,  
노트북 셀 단위로만 볼 때는 각 검색 방법이 따로 노는 느낌이 있었는데  
UI로 묶으니 실제 서비스 프로토타입처럼 보였기 때문이다.

사용자 질문 하나를 기준으로

- FAISS top 결과
- BM25 top 결과
- Hybrid top 결과

를 동시에 비교하니까  
어떤 방식이 어떤 질문에서 강한지 훨씬 빠르게 파악할 수 있었다.

#### 2. 검색 이력까지 남기니 디버깅하기 좋아졌다

검색 이력을 따로 저장해서 확인하는 구조도 들어갔는데,  
이 부분은 작지만 꽤 실용적으로 느껴졌다.

검색 시스템을 튜닝하다 보면  
어떤 질의에서 결과가 이상했는지를 다시 봐야 하는 경우가 많은데,  
이력이 있으면 반복 실험과 비교가 훨씬 쉬워진다.

즉 retrieval 프로젝트도  
모델만이 아니라 실험 UI와 관찰 도구까지 같이 있어야  
개선 속도가 더 빨라질 것 같았다.

---

## 정리

4주차 5일차는 retrieval 이론을 작은 프로젝트 형태로 묶어보는 시간이었다.

특히 아래 내용이 핵심으로 남았다.

1. 검색 프로젝트는 도메인 문서셋, 사용자 페르소나, 평가 질의셋을 같이 설계해야 한다.
2. Hit Rate, MRR, NDCG를 함께 보면 검색 품질을 더 입체적으로 볼 수 있다.
3. BM25와 FAISS는 강점이 다르기 때문에 hybrid search가 실전적으로 더 유용할 수 있다.
4. 동의어 확장과 custom multi-query 같은 도메인 튜닝이 성능 개선에 꽤 중요하다.
5. 비교 UI와 검색 이력까지 붙이면 실험과 디버깅이 훨씬 쉬워진다.

개인적으로는 이번 실습을 하면서  
이제 retrieval 공부가 단순 개념 정리 단계에서 조금씩 벗어나  
데이터셋 설계, 평가셋 작성, 튜닝, UI 확인까지 이어지는  
작은 시스템 만들기 단계로 넘어가고 있다는 느낌을 받았다.

앞으로 프로젝트를 더 진행할 때도  
모델 성능만 보지 말고  
어떤 사용자 질문을 기준으로, 어떤 평가셋으로, 어떤 UI에서 확인할 것인지까지  
같이 설계하는 습관을 가져가야 할 것 같다.
