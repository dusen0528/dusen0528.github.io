# 인프라 스터디 1주차

## 1주차

이번 1주차는 NGINX, Docker 같은 도구를 바로 깊게 파기보다  
"서비스 요청이 실제로 어떻게 흐르는지"를 운영체제 관점으로 다시 정리하는 시간이었다.

학부, 부트캠프, 실무를 거치면서 여러 도구를 써보긴 했지만,  
돌아가게 만드는 데 집중하다 보니 프로세스, 포트, 권한, 환경변수 같은 기초를 섞어서 이해하던 순간이 많았다.

이번 주의 흐름을 한 줄로 정리하면 아래와 같다.

```text
사용자 요청
-> 프록시/로드밸런서 유입
-> 애플리케이션 프로세스 처리
-> 로그/메트릭 기록
-> 장애 대응 및 복구
```

---

### 서비스 요청 흐름과 인프라 관점

#### 1. 요청은 어디서 시작해서 어디로 가는가

사용자 요청은 브라우저나 앱에서 시작해 네트워크를 거쳐 서버로 들어오고,  
앞단의 프록시/로드밸런서가 어떤 백엔드로 전달할지 결정한다.

이때 중요한 건 "누가 요청을 처음 받고, 어디로 넘기며, 최종적으로 어느 프로세스가 처리하는가"를 단계별로 명확히 보는 것이다.

작은 서비스에서는 애플리케이션이 직접 요청을 받을 수도 있지만,  
대부분은 NGINX 같은 프록시 계층이 먼저 트래픽을 받는다.

#### 2. 운영에서 중요한 건 정상 응답만이 아니다

요청이 처리되고 응답이 나가는 것만 보면 시스템이 잘 돌아가는 것처럼 보인다.  
하지만 운영에서는 로그와 메트릭이 함께 남아야 나중에 병목이나 장애 원인을 추적할 수 있다.

즉 인프라는 "정상 경로"를 이해하는 것과 동시에,  
"비정상 상황에서 복구 가능한 구조인지"를 보는 영역이라는 점이 더 선명해졌다.

---

### 리눅스, 프로세스, 포트 기초

#### 1. 파일과 프로세스는 다르다

디스크에 있는 실행 파일은 아직 서버가 아니다.  
운영체제가 메모리에 적재하고 PID를 부여해야 비로소 실행 중인 프로세스가 된다.

이 관점이 잡히고 나니, "서버가 떴다/죽었다"는 표현이 결국 프로세스 상태를 말한다는 점이 명확해졌다.

#### 2. 자주 보는 명령어의 목적

아래 명령어들은 단순 암기보다 "운영 질문에 답하기 위한 도구"로 이해하는 게 좋았다.

- `netstat -tnlp`: 어떤 프로세스가 어떤 포트를 리슨 중인지 확인
- `top`: 현재 자원 사용량(CPU/메모리)과 프로세스 상태 확인
- `tail -f <파일명>`: 애플리케이션 로그를 실시간 추적
- `df -h`: 디스크 사용량 확인
- `kill -15 <PID>` / `kill -9 <PID>`: 정상 종료 요청과 강제 종료 구분

특히 `kill -9`는 즉시 중단이라 마지막 수단으로 써야 한다는 점을 다시 확인했다.

#### 3. 포트는 "서비스의 입구"

포트는 어떤 프로세스가 네트워크 연결을 기다리는 번호이고,  
포트 충돌은 같은 번호를 두 프로세스가 동시에 점유하려 할 때 발생한다.

기본 포트 예시는 아래와 같다.

- HTTP: `80`
- HTTPS: `443`
- SSH: `22`

결국 핵심은 "현재 이 서버에서 누가 어떤 포트를 잡고 있는가"를 정확히 파악하는 습관이다.

---

### 파일 권한과 설정 분리

#### 1. 권한은 동작 문제이자 보안 문제

배포 과정에서 실행 권한이 없어 서비스가 기동되지 않거나,  
반대로 파일 권한을 과하게 열어 보안 리스크를 만드는 상황을 실제로 자주 겪었다.

예전에는 급할 때 `chmod 777`을 쓰곤 했는데,  
이번 정리에서 그 습관이 운영 안정성과 보안 모두에 좋지 않다는 점을 다시 확인했다.

기본 권한 비트는 아래처럼 기억해두면 실무에서 빠르게 해석할 수 있다.

| 구분 | r    | w    | x    |
| ---- | ---- | ---- | ---- |
| 의미 | 읽기 | 쓰기 | 실행 |
| 숫자 | 4    | 2    | 1    |

#### 2. 환경 변수와 설정 분리

설정값을 코드에 하드코딩하지 않고 환경 변수로 분리하면,  
같은 프로그램도 `local/dev/prod`에서 다르게 운영할 수 있다.

```bash
export TEMP_TOKEN=xxxx...
echo $TEMP_TOKEN
```

다만 `export`만으로는 현재 세션에서만 유효하므로,  
지속 적용하려면 `~/.bashrc` 또는 `~/.zshrc`에 반영하고 `source`로 로드해야 한다.

## 1주차 회고

이번 주를 지나며 가장 크게 느낀 점은,  
인프라는 도구 이름을 많이 아는 것보다 "흐름을 끊기지 않게 이해하는 것"이 더 중요하다는 것이다.

특히 아래 연결이 머릿속에서 하나의 그림으로 묶인 게 가장 큰 수확이었다.

```text
프로그램 파일
-> 프로세스 실행
-> 포트 리슨
-> 요청 처리
-> 로그/메트릭 기록
-> 장애 대응과 복구
```
