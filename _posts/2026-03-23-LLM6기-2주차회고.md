# 모두의 연구소 LLM 6기 재직자 과정 2주차 복습 및 회고

## 2주차

2주차는 1주차보다 훨씬 더 RAG에 가까워진 주차였다.

1주차에는 API 호출, LangChain, LCEL, 간단한 FAQ 챗봇까지 맛봤다면  
이번 주에는 그 챗봇이 왜 문서를 찾고, 왜 chunk를 나누고, 왜 embedding을 만들고, 왜 retriever가 필요한지를 처음부터 끝까지 한 흐름으로 배웠다.

이번 주의 큰 흐름을 한 줄로 정리하면 아래와 같다.

```text
문서
-> token / chunk로 나누기
-> embedding으로 벡터화
-> similarity로 관련 문서 찾기
-> vectorstore / retriever에 저장
-> LLM prompt에 context로 넣기
-> safe fallback으로 안전장치 두기
```

### Tokenizing, Chunking, Embeddings

> 1. Tokenizing과 BPE

LLM은 사람이 문장을 읽는 것처럼 한 덩어리로 이해하는 것이 아니라, 먼저 token이라는 작은 단위로 잘라서 처리한다.

가장 단순하게는 띄어쓰기 기준으로 나눌 수 있지만, 실제 LLM은 BPE(Byte Pair Encoding) 같은 subword 기반 방식을 사용한다.

핵심은 "자주 같이 나오는 조각을 하나로 묶는다"는 것이다.

즉 BPE는 의미를 이해하는 기술이 아니라, 텍스트를 더 효율적으로 쪼개기 위한 방식에 가깝다.

```python
import tiktoken

def tokenize_text(text, model="gpt-4o-mini"):
    enc = tiktoken.encoding_for_model(model)
    return enc.encode(text)

def count_tokens(text, model="gpt-4o-mini"):
    return len(tokenize_text(text, model=model))
```

이 부분에서 제일 먼저 느낀 점은, 평소 우리가 생각하던 "글자 수"나 "단어 수"와 실제 LLM이 보는 "token 수"는 완전히 다르다는 것이었다.

비용과 context 길이 제한이 결국 token 기준으로 움직이기 때문에, 이 후부터는 문장을 볼 때도 자연스럽게 "이건 몇 token쯤 될까?"를 생각하게 되었다.

> 1. Chunking과 Splitter

token이 모델이 실제로 읽는 아주 작은 단위라면, chunk는 긴 문서를 검색과 저장에 맞게 잘라놓은 큰 덩어리다.

즉 token과 chunk는 비슷해 보여도 역할이 다르다.

- token: 모델 내부 처리 단위
- chunk: 문서를 검색하기 좋게 자른 단위

긴 문서를 한 번에 LLM에 넣기 어렵기 때문에, 적당한 크기로 나누고 일부 overlap을 둬서 문맥이 경계에서 끊기지 않도록 한다.

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

def split_text(text, chunk_size=50, chunk_overlap=10):
    splitter = RecursiveCharacterTextSplitter.from_tiktoken_encoder(
        model_name="gpt-4o-mini",
        chunk_size=chunk_size,
        chunk_overlap=chunk_overlap,
    )
    return splitter.split_text(text)
```

처음에는 "굳이 overlap까지 둬야 하나?" 싶었는데, 실제로는 문장이 chunk 경계에서 끊기면 검색 결과가 꽤 어색해질 수 있다는 점이 인상적이었다.

RAG에서 chunking은 그냥 전처리가 아니라, 검색 품질을 좌우하는 중요한 설계 포인트라는 걸 느꼈다.

> 1. Embedding과 similarity

이번 주부터는 텍스트를 사람이 읽는 문자열이 아니라, 숫자 벡터로 다루기 시작했다.

embedding은 문장을 벡터 공간의 한 점으로 바꾸는 작업이라고 이해하면 편했다.

그리고 질문 벡터와 문서 벡터 사이의 cosine similarity를 계산해서 어떤 문서가 가장 비슷한지 찾는다.

```python
from sklearn.metrics.pairwise import cosine_similarity
from langchain_openai import OpenAIEmbeddings

embeddings_model = OpenAIEmbeddings(model="text-embedding-3-small")

def find_best_doc(query, docs):
    query_vector = embeddings_model.embed_query(query)
    doc_vectors = embeddings_model.embed_documents(docs)

    similarity = cosine_similarity(query_vector.reshape(1, -1), doc_vectors)
    best_index = similarity[0].argmax()

    return docs[best_index]
```

처음에는 벡터 숫자 하나하나를 보려고 했는데, 그건 큰 의미가 없었다. 중요한 건 숫자 자체가 아니라 "벡터끼리 얼마나 가까운가"였다.

이 부분이 익숙해지고 나서야, 왜 keyword 검색이 아니라 similarity 검색이 필요한지 조금씩 감이 잡히기 시작했다.

> 1. 임베딩 기반 분류와 CacheBackedEmbeddings

단순히 가장 비슷한 문서를 찾는 것뿐 아니라, 입력 문장이 어느 카테고리에 가까운지도 embedding으로 분류할 수 있었다.

```python
def classify(text, categories):
    text_emb = embeddings_model.embed_query(text)

    best_category = None
    best_score = -1

    for category, examples in categories.items():
        example_embs = embeddings_model.embed_documents(examples)
        scores = cosine_similarity([text_emb], example_embs)[0]
        avg_score = scores.mean()

        if avg_score > best_score:
            best_score = avg_score
            best_category = category

    return best_category
```

이 부분이 흥미로웠던 건, 분류를 꼭 if문이나 정규식으로만 하는 게 아니라  
"가장 비슷한 카테고리 예시들과 얼마나 가까운가"로도 볼 수 있다는 점이었다.

그리고 `CacheBackedEmbeddings`를 배우면서, 같은 문서를 매번 다시 임베딩하는 건 비효율적이기 때문에 cache로 재사용하는 감각도 같이 익혔다.

즉 이 시점부터는 단순히 "임베딩이 뭔가"가 아니라, "임베딩을 반복해서 쓸 때는 어떻게 관리할 것인가"까지 고민하게 되었다.

> 1. Batch Embeddings와 VectorStore

문서가 많아지면 한 번에 전부 처리하기보다 batch 단위로 나눠 임베딩하는 편이 더 관리하기 쉽다.

```python
def process_documents(docs, embeddings_model, batch_size=5):
    all_vectors = []

    for i in range(0, len(docs), batch_size):
        batch = docs[i:i + batch_size]
        vectors = embeddings_model.embed_documents(batch)
        all_vectors.extend(vectors)

    return all_vectors
```

여기서 `append`가 아니라 `extend`를 써야 결과가 중첩되지 않는다는 점도 실전에서 꽤 중요한 포인트였다.

또한 vectorstore를 처음 제대로 접하면서, 지금까지 직접 cosine similarity를 계산하던 흐름이 "검색 가능한 저장소에 임베딩을 넣고 꺼내는 구조"로 바뀌기 시작했다.

```python
from langchain_community.vectorstores import FAISS

vectorstore = FAISS.from_texts(faq_docs, embeddings_model)
results = vectorstore.similarity_search_with_score("RAG가 뭐에요?", k=2)
```

이때부터는 "임베딩 벡터 자체"보다 "벡터를 저장하고 검색하는 시스템"으로 시야가 확장되었다.

### class, DocumentStore, RAG Basics

> 1. class, WordCounter, DocumentStore

0319 수업에서는 갑자기 class가 등장해서 처음에는 왜 여기서 class를 배우는지 조금 뜬금없게 느껴졌다.

하지만 다시 복기해보니, 결국 RAG는 문서 목록, 출처, 검색 결과, 임베딩 모델 등 여러 상태를 함께 관리해야 하기 때문에 class 흐름이 필요했다.

```python
class WordCounter:
    def __init__(self):
        self.text = ""

    def add_text(self, text):
        if self.text:
            self.text += " "
        self.text += text

    def count_words(self):
        return len(self.text.split())
```

`WordCounter`는 아주 작은 예제였지만, 객체가 상태를 누적한다는 감각을 익히기에 좋았다.

이후 `DocumentStore`로 넘어가면서 문자열 하나를 다루는 구조가, 여러 문서를 저장하고 검색하는 구조로 확장되었다.

```python
class DocumentStore:
    def __init__(self):
        self.documents = []
        self.next_id = 1
        self.embeddings = embeddings_model

    def add(self, content, source="unknown"):
        self.documents.append({
            "id": self.next_id,
            "content": content,
            "source": source,
        })
        self.next_id += 1

    def search(self, keyword):
        return [doc for doc in self.documents if keyword in doc["content"]]
```

여기서 중요한 건 `search()`와 `retrieve()`의 차이였다.

- `search()`는 문자열 포함 여부 비교
- `retrieve()`는 embedding similarity 기반 의미 비교

즉 검색이라고 다 같은 검색이 아니고, keyword 기반 검색과 semantic search는 완전히 다른 방식이라는 점을 더 선명하게 이해하게 되었다.

그리고 여기서부터 RAG의 핵심 구조인 `retrieve + generate`가 비로소 납득되기 시작했다.

문서를 찾기만 하고 LLM에 안 넣으면 그냥 검색이고,  
질문만 LLM에 보내면 그냥 일반 호출이다.

검색 결과를 실제 생성 단계에 연결해야 비로소 RAG가 된다.

### Document 로딩과 metadata

> 1. LangChain Document와 파일 로딩

수업 후반부에서는 txt, csv, json 파일을 `Document` 객체로 바꾸는 실습도 했다.

처음에는 그냥 문자열로만 다뤄도 되는 줄 알았는데, 실제로는 `page_content`와 `metadata`를 함께 관리하는 게 훨씬 중요했다.

```python
from langchain_core.documents import Document

doc = Document(
    page_content="문서 내용",
    metadata={"source": "file.txt", "file_type": "txt"}
)
```

txt는 파일 단위, csv는 row 단위, json은 직렬화된 문자열 단위로 문서화할 수 있었다.

```python
def load_csv_file(file_path):
    documents = []
    path = Path(file_path)

    with open(path, "r", encoding="utf-8") as f:
        reader = csv.DictReader(f)
        for i, row in enumerate(reader):
            content = " | ".join(f"{k}: {v}" for k, v in row.items())
            documents.append(
                Document(
                    page_content=content,
                    metadata={
                        "source": path.name,
                        "file_type": "csv",
                        "row_index": i,
                    },
                )
            )
    return documents
```

이 부분을 배우고 나서야, 나중에 검색 결과를 보여줄 때 왜 source나 row_index 같은 정보가 필요한지 확실히 이해됐다.

즉 `metadata`는 단순 부가 정보가 아니라, 검색 결과를 설명 가능하게 만들어주는 중요한 요소였다.

### FAISS, Retriever, LCEL RAG

> 1. FAISS와 Retriever

0320부터는 이제 진짜 RAG의 형태가 나오기 시작했다.

먼저 문서를 section 단위로 나눈 뒤 `Document`로 만들고, 그걸 FAISS에 저장한다.

```python
def build_documents(sample_dir):
    documents = []

    files = {
        sample_dir / "company_policy.txt": "사내규정",
        sample_dir / "ai_report.txt": "AI보고서",
    }

    for fpath, category in files.items():
        text = fpath.read_text(encoding="utf-8")
        for section in text.strip().split("\n\n"):
            if section.strip():
                documents.append(
                    Document(
                        page_content=section.strip(),
                        metadata={"category": category, "source": fpath.name},
                    )
                )
    return documents
```

그 다음 vectorstore를 만들고 retriever를 연결한다.

```python
vectorstore = FAISS.from_documents(documents, embeddings)
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})
```

이제부터는 질문을 넣으면 관련 문서를 꺼내오는 검색기가 생긴 것이다.

이 부분에서 특히 조심해야 했던 건 `similarity_search_with_score()`의 score 해석이었다.  
cosine similarity처럼 "높을수록 좋다"가 아니라, FAISS는 거리 개념이라 "작을수록 가깝다"로 읽어야 할 수 있었다.

이걸 반대로 이해하면 threshold 조건이 완전히 뒤집혀버리기 때문에 생각보다 중요했다.

> 1. LCEL RAG chain과 safe fallback

이번 주 후반부에서 제일 재밌었던 부분은, 지금까지 따로 배우던 retriever와 prompt, llm, parser가 하나의 체인으로 연결되는 순간이었다.

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

def format_docs(docs):
    return "\n--\n".join(
        f"[{d.metadata.get('category', '')}] {d.page_content}"
        for d in docs
    )

def build_rag_chain(retriever, llm):
    rag_prompt = ChatPromptTemplate.from_messages([
        ("system", "사내 도우미 챗봇입니다. 참고문서:\n{context}\n\n문서 기반으로 답변해주세요."),
        ("user", "{question}"),
    ])

    rag_chain = (
        {
            "context": retriever | format_docs,
            "question": RunnablePassthrough(),
        }
        | rag_prompt
        | llm
        | StrOutputParser()
    )
    return rag_chain
```

이 구조를 보고 나서야, RAG가 단순히 "문서 검색하고 답변한다"가 아니라 아래와 같은 꽤 명확한 파이프라인이라는 게 이해됐다.

```text
질문
-> retriever가 관련 문서 검색
-> format_docs로 context 문자열 생성
-> prompt에 context와 question 주입
-> llm이 답변 생성
-> parser가 최종 문자열 반환
```

그리고 safe fallback도 꽤 중요했다.

검색 결과가 충분히 관련 있지 않으면, 그냥 억지로 답변을 생성하지 않고 `"해당 정보가 없습니다"`처럼 막는 전략이다.

```python
def verify_query(question, vectorstore, threshold=1.3, k=3):
    results = vectorstore.similarity_search_with_score(question, k=k)

    best_doc = None
    best_score = float("inf")

    for doc, score in results:
        if score < best_score:
            best_score = score
            best_doc = doc

    if best_score > threshold:
        return "해당 정보가 없습니다"

    return best_doc.page_content
```

이 부분을 보면서, RAG도 검색이 틀리면 엉뚱한 문서를 근거로 답변할 수 있기 때문에, 검색 품질과 안전장치가 정말 중요하다는 걸 느꼈다.

## 2주차 회고

2주차는 1주차보다 확실히 더 어려웠다.

1주차는 "LLM을 어떻게 호출하는가"에 가까웠다면, 2주차는 "LLM이 외부 문서를 어떻게 참고하게 만들 것인가"에 더 가까웠기 때문이다.

처음에는 token, chunk, embedding, similarity, cache, vectorstore, retriever, LCEL이 전부 따로 노는 개념처럼 느껴졌다.

그런데 복기하면서 정리해보니 결국 모두 아래 흐름으로 이어지고 있었다.

```text
문서를 잘게 나눈다
-> 나눈 문서를 숫자 벡터로 바꾼다
-> 질문과 가장 가까운 문서를 찾는다
-> 그 문서를 LLM prompt에 넣는다
-> 관련 없으면 안전하게 fallback한다
```

개인적으로는 `Document`, `retriever`, `format_docs`, `safe fallback` 부분이 특히 인상 깊었다.

그냥 "LLM에게 질문한다"에서 끝나는 게 아니라,  
"어떤 문서를 근거로 답하게 만들고, 그 문서가 충분히 관련 있는지까지 검증한다"는 흐름이 훨씬 실무적으로 느껴졌다.

이번 주를 지나며 느낀 건, RAG는 생각보다 복잡한 기술이라기보다  
검색과 생성 사이를 잘 연결하는 설계 문제에 더 가깝다는 것이다.

3주차에는 이 흐름이 더 익숙해지길 기대해본다.
