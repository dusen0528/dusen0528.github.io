# 모두의 연구소 LLM 6기 재직자 과정 4주차 2일차 복습 및 정리

## 4주차 2일차

4주차 2일차는 전날 배운 Hit@k에서 한 걸음 더 나아가  
retrieval을 더 다양한 지표로 평가하고,  
생성 답변은 BLEU로 비교하고,  
마지막에는 LLM을 평가자로 사용하는 방식까지 다룬 날이었다.

전날이 "정답 문서를 포함시켰는가"를 보는 입문이었다면,  
오늘은 "얼마나 정확하게 찾았는가", "얼마나 놓치지 않았는가",  
"생성 결과를 자동 평가할 수 있는가"까지 확장한 느낌이었다.

이번 수업의 큰 흐름을 한 줄로 정리하면 아래와 같다.

```text
documents / qa_dataset 준비
-> search_results_cache 생성
-> Precision@k / Recall@k / F1@k / R-precision / F-beta 계산
-> query별 표와 평균 표 확인
-> BLEU로 생성 답변 자동 비교
-> LLM-as-Judge로 컨텍스트 관련성 평가
```

이번 수업을 지나고 나니,  
검색과 생성을 평가하는 방식은 하나가 아니라  
목적에 따라 서로 다른 지표를 함께 봐야 한다는 점이 더 선명해졌다.

---

### Retrieval Metric 확장

#### 1. Precision@k

가장 먼저 구현한 지표는 `precision_at_k`였다.

```python
def precision_at_k(query_id, k):
    qa = next(q for q in qa_dataset if q['query_id']== query_id)
    relevant_ids = set(qa['relevant_doc_ids'])
    retrieved = search_results_cache[query_id][:k]
    retrieved_ids = {r['doc_id'] for r in retrieved}

    relevant_retrieved = relevant_ids & retrieved_ids
    return len(relevant_retrieved) / k
```

이 지표는 top-k 결과 중 실제 정답 문서가 얼마나 들어 있었는지를 본다.

예를 들어 top-3 중 정답이 1개면 `P@3 = 1/3`이다.

즉 precision은  
"가져온 결과가 얼마나 깔끔한가"를 보는 지표라고 이해하면 편했다.

정답이 적고 관련 없는 문서가 많이 섞이면 precision이 낮아진다.

#### 2. Recall@k

반대로 `recall_at_k`는 정답 문서를 얼마나 빠뜨리지 않았는지를 본다.

```python
def recall_at_k(query_id, k):
    qa = next(q for q in qa_dataset if q['query_id']== query_id)
    relevant_ids = set(qa['relevant_doc_ids'])
    retrieved = search_results_cache[query_id][:k]
    retrieved_ids = {r['doc_id'] for r in retrieved}
    
    relevant_retrieved = relevant_ids & retrieved_ids
    return len(relevant_retrieved) / len(relevant_ids) if relevant_ids else 0.0
```

이 지표는 특히 정답 문서가 여러 개인 질문에서 중요했다.

- 전체 정답이 2개인데 1개만 찾았으면 `R@k = 0.5`
- 둘 다 찾았으면 `R@k = 1.0`

즉 recall은  
"놓치지 않고 얼마나 많이 찾았는가"에 더 가깝다.

---

### F1, F-beta, R-precision

#### 1. precision만 높아도 안 되고 recall만 높아도 안 된다

검색 결과가 아주 깔끔해도 정답을 많이 놓치면 문제고,  
반대로 정답을 대부분 포함해도 관련 없는 문서가 너무 많이 섞이면 역시 아쉽다.

그래서 둘의 균형을 보는 F계열 지표가 등장했다.

```python
def f1_at_k(query_id, k):
    p = precision_at_k(query_id, k)
    r = recall_at_k(query_id, k)
    return 2 * p * r / (p + r)
```

F1은 precision과 recall을 같은 비중으로 보는 조화평균이다.

#### 2. F-beta

그리고 `F-beta`는 어느 쪽을 더 중요하게 볼지 조절할 수 있다.

```python
def f_beta_at_k(query_id, k , beta):
    p = precision_at_k(query_id, k)
    r = recall_at_k(query_id, k)

    beta_sq = beta ** 2
    if beta_sq * p + r == 0:
        return 0
    return (1+beta_sq) * p * r / (beta_sq * p + r)
```

해석은 아래처럼 정리할 수 있었다.

- `beta = 0.5`면 precision 쪽을 더 중요하게 보고
- `beta = 1.0`이면 precision과 recall을 균형 있게 보고
- `beta = 2.0`이면 recall 쪽을 더 중요하게 본다

이 부분이 좋았던 건,  
같은 retrieval 문제라도 서비스 목적에 따라 중요하게 볼 지표가 달라질 수 있다는 점을 느끼게 해줬다는 것이다.

예를 들어 FAQ 검색이라면 precision이 중요할 수 있고,  
근거 문서를 최대한 빠짐없이 모아야 하는 태스크라면 recall이 더 중요할 수 있다.

#### 3. R-precision

`R-precision`도 새로 등장했다.

```python
def r_precision(query_id):
    qa = next(q for q in qa_dataset if q['query_id']== query_id)
    relevant_ids = set(qa['relevant_doc_ids'])

    R = len(relevant_ids)

    if R==0:
        return 0.0
    
    return precision_at_k(query_id, R)
```

정답 문서 수가 `R`개라면  
top-R 결과에서 precision을 계산하는 방식이다.

질문마다 정답 문서 개수가 다를 때도  
조금 더 공정하게 볼 수 있다는 점이 흥미로웠다.

---

### K 값에 따른 변화

#### 1. K=1~5 표를 보는 이유

이번 수업에서는 `K=1~5` 구간에서 평균 precision, recall, F-beta를 표로 비교했다.

이 과정에서 느낀 가장 중요한 점은  
K를 늘리면 항상 좋아지는 것이 아니라 지표별로 움직임이 다르다는 점이었다.

대체로 아래 같은 흐름이 나온다.

- `precision@k`는 k가 커질수록 떨어지거나 유지되기 쉽고
- `recall@k`는 k가 커질수록 올라가거나 유지되기 쉽고
- `F-beta`는 어떤 쪽을 더 중시하느냐에 따라 결과 해석이 달라진다

즉 top-1은 아주 정확한지,  
top-3이면 충분한지,  
top-5까지 넓혀야 놓치는 문서가 줄어드는지를 함께 봐야 한다.

이런 비교를 하면서 retrieval도 단일 점수 하나로 끝나는 게 아니라  
컷오프와 목적에 따라 여러 관점에서 봐야 한다는 점이 인상 깊었다.

### BLEU

#### 1. 생성 답변 자동 비교

검색 평가 다음으로는 생성 평가 메트릭인 BLEU를 다뤘다.

```python
def bleu_score(reference, hypothesis, max_n):
    ref_tokens = reference.split()
    hyp_tokens = hypothesis.split()

    c, r = len(hyp_tokens), len(ref_tokens)
    bp = 1.0 if c >=r else np.exp(1-r/c) if c>0 else 0.0
```

BLEU는 참조 답변과 생성 답변의 n-gram 겹침 정도를 보고 점수를 계산한다.

핵심 아이디어는 단순하다.

1. reference와 hypothesis를 토큰화하고
2. 1-gram부터 n-gram까지 겹침 비율을 보고
3. 너무 짧은 답변에는 brevity penalty를 준다

즉 완전히 같은 문장이 아니어도  
표현이 얼마나 비슷한지 어느 정도 자동으로 비교할 수 있다.

#### 2. clipped precision과 brevity penalty

BLEU에서 중요했던 포인트는 두 가지였다.

- clipped precision: 특정 표현을 과하게 반복해서 점수를 부풀리는 걸 막는다
- brevity penalty: 너무 짧은 답변이 부당하게 높은 점수를 받지 않게 한다

이 구조를 보면서 자동 평가 메트릭도  
단순 문자 비교보다 훨씬 더 많은 예외를 고려하고 있다는 걸 알게 되었다.

하지만 동시에 한계도 분명했다.

같은 의미를 다른 표현으로 말하면  
n-gram 겹침이 낮아져서 점수가 낮게 나올 수 있기 때문이다.

즉 BLEU는 빠르고 편리하지만,  
의미적 유사성을 완벽하게 반영하는 도구는 아니라는 점을 같이 배웠다.

### LLM-as-Judge

#### 1. LLM을 평가자로 쓰는 방식

마지막에는 LLM을 평가자로 사용하는 방식도 다뤘다.

이번 노트북에서는 질문과 검색된 컨텍스트를 보여주고,  
각 컨텍스트가 질문과 얼마나 관련 있는지 1~5점으로 평가하게 했다.

```python
def evaluate_context_relevance(question, contexts, model):
    context_text = '\n\n'.join([
        f"[컨텍스트 {i+1}]: {ctx}" for i , ctx in enumerate(contexts)
    ])

    prompt = f"""
    질문과 검색된 컨텍스트의 관련성을 평가하세요
    ...
    ## 응답 형식(JSON)
    {{"scores": [점수1, 점수2 , ....], "average": 평균점수, "reasoning": "평가 근거"}}
    """
```

이 접근의 장점은 꽤 분명했다.

- 사람이 일일이 채점하지 않아도 되고
- 단순 문자열 겹침보다 의미 기반 평가가 가능하고
- 컨텍스트 품질을 정성적으로 확인하기 좋다

그리고 JSON 형식으로 강제해서 점수와 근거를 함께 받는 방식도 인상적이었다.

단순히 "좋다/나쁘다"가 아니라  
왜 그렇게 평가했는지 reasoning까지 남길 수 있기 때문이다.

#### 2. 자동 메트릭과 LLM Judge는 서로 대체재가 아니라 보완재다

이날 수업을 들으며 느낀 건,  
BLEU 같은 자동 메트릭과 LLM-as-Judge는 경쟁 관계라기보다 보완 관계에 가깝다는 점이었다.

- BLEU는 빠르고 반복 가능하지만 표현 차이에 약하고
- LLM Judge는 의미 기반으로 더 유연하지만 프롬프트와 모델에 영향을 받는다

즉 실제 프로젝트에서는 하나만 보기보다  
retrieval metric, generation metric, judge 기반 정성 평가를 함께 보는 쪽이 더 현실적일 것 같았다.

---

## 정리

4주차 2일차는 검색과 생성 평가를 더 입체적으로 보기 시작한 날이었다.

특히 아래 내용이 핵심으로 남았다.

1. retrieval은 Precision@k, Recall@k, F1@k, R-precision, F-beta처럼 여러 관점으로 봐야 한다.
2. BLEU는 생성 답변 자동 비교에 유용하지만 의미적 유사성을 완벽히 반영하지는 못한다.
3. LLM-as-Judge는 사람이 하던 정성 평가 일부를 자동화하는 현실적인 방법이 될 수 있다.

개인적으로는 이번 수업을 통해  
"평가는 정답 하나와 비교해서 끝내는 것"이라는 생각에서 벗어나게 되었다.

RAG 시스템을 제대로 보려면

- 검색 단계는 검색 지표로 보고
- 생성 단계는 생성 메트릭으로 보고
- 필요하면 LLM Judge나 사람 평가까지 붙이는 식으로

레이어별로 평가를 나눠서 생각해야 한다는 점이 점점 더 명확해지고 있다.
