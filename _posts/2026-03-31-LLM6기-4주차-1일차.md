# 모두의 연구소 LLM 6기 재직자 과정 4주차 1일차 복습 및 정리

## 4주차 1일차

4주차 1일차는 RAG를 더 잘 만드는 법보다는,  
내가 만든 검색기가 실제로 잘 동작하고 있는지 어떻게 검증하고 평가할지를 배우는 시간에 가까웠다.

그동안은 문서를 넣고 질문해보고 답변이 그럴듯하면 넘어가는 경우가 많았는데,  
이번에는 documents와 qa_dataset을 직접 만들고,  
검색 결과를 cache로 저장한 뒤 Hit@k 같은 지표로 확인하는 흐름을 처음부터 끝까지 다뤘다.

이번 수업의 큰 흐름을 한 줄로 정리하면 아래와 같다.

```text
documents 만들기
-> qa_dataset 만들기
-> FAISS 기반 검색기 구성
-> 질문별 검색 결과 cache 저장
-> dataset 정합성 검증
-> Hit@k 계산
-> 문서/질문 추가 후 다시 평가
```

이날 가장 크게 느낀 점은,  
RAG는 검색기를 만드는 것도 중요하지만 그보다 먼저 "평가 가능한 데이터셋"을 만드는 일이 정말 중요하다는 점이었다.

---

### documents와 qa_dataset

#### 1. documents는 단순 텍스트 모음이 아니다

이번 노트북에서는 문서를 아래와 같은 형태로 관리했다.

```python
{
    "doc_id": "doc_001",
    "title": "트랜스포머 아키텍처",
    "content": "트랜스포머는 2017년 'Attention is All You Need' 논문에서 제안된 아키텍처입니다."
}
```

여기서 중요했던 건 `content`만 있는 것이 아니라  
`doc_id`, `title`까지 함께 있다는 점이었다.

- `doc_id`는 검색 결과와 정답 문서를 연결하는 기준 키가 되고
- `title`은 사람이 검색 결과를 읽고 해석하기 쉽게 만들어 주며
- `content`는 실제 검색 대상 본문이 된다

즉 문서셋은 그냥 문자열 리스트가 아니라  
"평가와 디버깅이 가능한 검색 코퍼스"로 관리해야 한다는 걸 느꼈다.

#### 2. qa_dataset은 검색 평가용 기준표다

질문 데이터도 단순 질문 모음이 아니라 아래처럼 정답 문서 id와 함께 구성되었다.

```python
{
    "query_id": "q01",
    "question": "트랜스포머의 핵심 메커니즘은 무엇인가요?",
    "relevant_doc_ids": ["doc_001"],
    "ground_truth": "트랜스포머의 핵심 메커니즘은 셀프 어텐션입니다."
}
```

이 구조를 보면서 확실히 이해된 건,  
검색 평가는 결국 "이 질문이면 어떤 문서가 나와야 하는가"가 미리 정해져 있어야 한다는 점이었다.

즉 qa_dataset은 단순 테스트 질문이 아니라,

- 질문
- 정답 문서 id
- 필요하면 참조 답변

을 같이 가진 라벨 데이터셋 역할을 한다.

---

### Vector Store와 Search Cache

#### 1. LangChain Document와 metadata

문서셋은 이후 LangChain `Document` 객체로 바꿔서 FAISS에 넣었다.

```python
langchain_docs = [
    Document(
        page_content=doc['content'],
        metadata={"doc_id": doc['doc_id'], 'title': doc['title']}
    ) for doc in documents
]

vectorstore = FAISS.from_documents(langchain_docs, embeddings)
```

여기서 배운 중요한 포인트는 metadata 설계였다.

검색 결과에서 다시 `doc_id`, `title`을 꺼내 쓰려면  
문서를 저장할 때부터 metadata에 함께 넣어둬야 한다.

예전에는 content만 있으면 된다고 생각했는데,  
실제로 평가하려면 문서 식별자와 설명 정보가 같이 있어야 훨씬 편하다는 걸 알게 되었다.

#### 2. search_results_cache

질문마다 top-k 검색 결과를 미리 저장하는 방식도 꽤 인상적이었다.

```python
search_results_cache = {}
for qa in qa_dataset:
    results = search_documents(qa['question'], k=5)
    search_results_cache[qa['query_id']] = results
```

이 구조를 쓰면

- 같은 질문에 대해 검색을 반복하지 않아도 되고
- 평가 함수를 단순하게 만들 수 있고
- query별 검색 결과를 쉽게 들여다볼 수 있다

즉 `search_results_cache`는  
"질문별 검색 결과 스냅샷" 같은 역할을 한다고 이해하면 편했다.

---

### Dataset Validation

#### 1. 평가 전에 데이터셋부터 검증해야 한다

후반부에서는 새 문서와 새 QA를 추가한 뒤,  
그 QA가 가리키는 `relevant_doc_ids`가 실제 문서셋 안에 존재하는지 먼저 확인했다.

```python
def validate_dataset(docs, qa_pairs):
    doc_ids = {d['doc_id'] for d in docs}
    errors = []
    for qa in qa_pairs:
        for rid in qa['relevant_doc_ids']:
            if rid not in doc_ids:
                errors.append(f"{qa['query_id']} : {rid} 문서 없음")
```

이 부분이 정말 중요하게 느껴졌던 이유는,  
검색기가 못 찾은 것인지 데이터 라벨이 잘못된 것인지 먼저 구분해야 하기 때문이다.

예를 들어 QA에는 `doc_012`가 정답이라고 써놨는데  
실제 문서셋에는 그 문서가 없다면  
그건 검색기 문제가 아니라 데이터 문제다.

즉 모델이나 검색기 성능을 보기 전에  
데이터셋 정합성을 먼저 확인하는 습관이 필요하다는 걸 배웠다.

#### 2. set을 쓰는 이유

이 함수에서 `set`을 사용하는 이유도 같이 복습했다.

```python
doc_ids = {d['doc_id'] for d in docs}
```

리스트보다 membership 검사 속도가 빠르고,  
"존재 여부 확인"이라는 목적에도 더 잘 맞는다.

작은 실습에서는 차이가 잘 안 느껴질 수 있지만,  
데이터가 커질수록 이런 기본기가 결국 중요해질 것 같았다.

---

### Hit@k

#### 1. 적어도 하나의 정답 문서를 top-k 안에 넣었는가

이날 핵심 평가 지표는 `Hit@k`였다.

```python
def hit_rate_at_k(query_id, k):
    qa = next(q for q in qa_dataset if q['query_id'] == query_id)
    
    relevant_ids = set(qa['relevant_doc_ids'])
    retrieved = search_results_cache[query_id][:k]
    retrieved_ids = {r['doc_id'] for r in retrieved}
    
    return 1 if relevant_ids & retrieved_ids else 0
```

의미는 아주 단순하다.

- 질문별 정답 문서 집합이 있고
- 검색 결과 top-k가 있을 때
- 둘의 교집합이 하나라도 있으면 성공

즉 `Hit@k`는  
"top-k 안에 정답 문서를 적어도 하나는 넣었는가"를 보는 가장 기본적인 retrieval 평가 지표다.

#### 2. Average Hit@k

질문 하나만 보면 운이 섞일 수 있기 때문에,  
전체 질문에 대해 평균을 내는 방식도 같이 사용했다.

```python
def average_hit_rate_at_k(k):
    hits = [hit_rate_at_k(qa['query_id'], k) for qa in qa_dataset]
    return sum(hits)/ len(hits)
```

이 값을 보면

- top-1만 봐도 잘 찾는지
- top-3 정도면 대부분 포함되는지
- top-5까지 넓힐 가치가 있는지

를 감으로가 아니라 숫자로 볼 수 있다.

이전까지는 "검색 결과가 대충 괜찮아 보인다" 정도였다면,  
이제는 "top-k 기준으로 얼마나 자주 정답을 포함시키는가"를 생각하게 되었다.

---

### 문서 확장과 재평가

#### 1. 문서셋이 바뀌면 vectorstore와 cache도 다시 만들어야 한다

후반부에는 새 문서와 새 질문이 추가되었는데,  
이때 단순히 목록만 붙이는 것으로 끝나지 않고 검색기와 cache도 다시 계산했다.

이 흐름이 당연해 보이지만 꽤 중요했다.

- 문서셋이 바뀌면 검색 공간 자체가 달라지고
- 그러면 유사도 검색 결과도 달라질 수 있고
- 그에 따라 기존 평가 결과도 달라질 수 있다

즉 RAG 실험에서는 데이터셋 변경과 검색기 재구성이 항상 같이 움직여야 한다는 점을 다시 확인했다.

---

## 정리

4주차 1일차는 지금까지 만들어온 RAG 흐름을 조금 더 실험 가능한 형태로 바꾸는 시간이었다.

특히 아래 세 가지가 핵심이었다.

1. documents와 qa_dataset을 평가 가능한 구조로 설계해야 한다.
2. 검색기 성능을 보기 전에 dataset validation으로 라벨 정합성을 먼저 확인해야 한다.
3. Hit@k 같은 retrieval metric으로 검색 품질을 숫자로 보기 시작해야 한다.

개인적으로는 이 수업을 통해  
RAG를 "잘 답하는 챗봇" 관점에서만 보던 시선이  
"검색 시스템을 어떻게 실험하고 측정할 것인가" 쪽으로 조금 넓어진 느낌이었다.

앞으로는 단순히 답변 결과만 보는 게 아니라,  
정답 문서가 몇 위에 나오는지, 어떤 질문에서 계속 miss가 나는지까지 같이 봐야  
실제로 검색 품질을 개선할 수 있을 것 같다.
