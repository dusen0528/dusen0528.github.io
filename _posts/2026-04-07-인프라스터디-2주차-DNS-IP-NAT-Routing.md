# DNS -> IP -> NAT -> Routing

이번 글은 SerahKim님이 정리해주신 네트워크 흐름 글을 읽고,  
그 내용을 다시 정리해보면서 추가로 궁금했던 부분을 더 공부해본 기록이다.

SerahKim님 글에서는 사용자가 `www.google.com` 같은 도메인을 입력했을 때,

```text
DNS 조회
-> 서버 IP 확인
-> 라우팅 테이블 확인
-> 기본 게이트웨이로 전송
-> NAT를 통해 공인 IP로 변환
-> 인터넷을 통해 서버까지 전달
```

라는 흐름으로 요청이 전달된다고 설명해주셨다.

DNS, IP, NAT, Routing이라는 개별 개념이 실제 요청 흐름 안에서 어떻게 이어지는지 한 번에 볼 수 있어서 전체 맥락을 이해하는 데 도움이 많이 됐다.

---

## SerahKim님 글을 읽고 다시 정리한 핵심 흐름

내가 이해한 내용을 한 줄로 정리하면 아래와 같다.

```text
사용자가 도메인 입력
-> DNS가 도메인을 IP로 변환
-> 운영체제가 라우팅 테이블 확인
-> 외부 목적지라면 기본 게이트웨이로 전달
-> 공유기가 NAT 수행
-> 공인 IP 기반으로 인터넷 구간을 지나 서버 도착
```

이 흐름에서 특히 중요한 포인트는 아래 네 가지였다.

- DNS는 도메인을 실제 통신 가능한 IP 주소로 변환한다.
- 외부 네트워크로 나갈 때는 라우팅 테이블을 보고 기본 게이트웨이로 보낸다.
- 공유기에서는 NAT를 통해 사설 IP를 공인 IP로 바꾼다.
- 서버는 최종적으로 공인 IP 기반 요청을 받고 응답한다.

여기까지가 큰 흐름이라면, 글을 읽으면서 나는 두 가지가 더 궁금해졌다.

- 기본 게이트웨이로 보낸다는 것은 실제로 어떤 방식으로 이루어질까?
- DNS 조회는 정말 매번 Root DNS부터 다시 시작할까?

이 두 부분을 추가로 찾아보며 정리한 내용은 아래와 같다.

---

## 1. 기본 게이트웨이로 바로 보내는 것이 아니라 ARP가 먼저 필요하다

우리는 보통 "라우팅 테이블을 확인한 뒤 기본 게이트웨이로 보낸다"라고 간단히 말한다.  
그런데 실제로 데이터를 보내려면 IP 주소만으로는 부족하고, 같은 네트워크 안에서 다음 목적지의 MAC 주소가 필요하다.

즉 외부 서버로 패킷을 보낼 때 내 컴퓨터는 곧바로 서버의 MAC 주소를 아는 것이 아니라,  
먼저 **기본 게이트웨이의 MAC 주소**를 알아내야 한다.

이때 사용하는 것이 `ARP(Address Resolution Protocol)`이다.

### ARP가 필요한 이유

- IP는 네트워크 상에서 목적지를 식별하는 주소다.
- MAC 주소는 같은 네트워크에서 실제 프레임을 전달할 때 필요한 주소다.
- 따라서 전송 직전에는 "다음 홉"의 MAC 주소를 알아내야 한다.

### 실제 흐름

```text
1. DNS로 목적지 서버 IP 확인
2. 라우팅 테이블 확인
3. 외부 네트워크라면 기본 게이트웨이 사용 결정
4. ARP로 기본 게이트웨이의 MAC 주소 조회
5. 게이트웨이 MAC 주소를 목적지로 하여 프레임 전송
```

즉 "기본 게이트웨이로 전송한다"는 말 안에는 사실 아래 과정이 숨어 있다.

```text
기본 게이트웨이로 보낸다
= 기본 게이트웨이의 MAC 주소를 ARP로 알아낸 뒤 전송한다
```

이 부분을 알고 나니, 네트워크 흐름이 IP 레벨에서만 움직이는 것이 아니라  
실제로는 링크 계층 정보까지 필요하다는 점이 더 명확하게 보였다.

## 2. DNS 조회는 매번 Root DNS부터 시작되지 않는다

보통 DNS는 아래와 같은 흐름으로 설명된다.

```text
클라이언트
-> ISP DNS
-> Root DNS
-> TLD DNS
-> Authoritative DNS
-> IP 응답
```

그런데 이 과정도 매번 처음부터 반복되는 것은 아니라고 한다.  
이유는 DNS 서버가 응답 결과를 **캐시**하기 때문이다.

### TTL이란?

`TTL(Time To Live)`은 DNS 응답을 얼마나 오래 캐시에 저장할 수 있는지를 나타내는 시간이다.

예를 들어 `www.google.com` 조회 결과의 TTL이 `195초`라면,

- DNS 서버는 그 응답을 최대 195초 동안 저장할 수 있고
- 같은 요청이 다시 들어오면 캐시된 값을 바로 응답할 수 있으며
- TTL이 0이 되면 캐시가 만료되어 다시 상위 DNS 서버를 따라가며 조회한다

TTL은 시간이 지날수록 줄어든다.

```text
처음 조회: TTL 195
다시 조회: TTL 190
다시 조회: TTL 55
TTL 0: 캐시 만료 -> 재조회
```

### DNS 캐시를 기준으로 보면

실제 동작은 아래처럼 이해하는 것이 더 자연스럽다.

```text
if (cache && TTL_valid)
    캐시된 응답 반환
else
    Root -> TLD -> Authoritative 순서로 재조회
```

즉 누군가가 다시 `www.google.com`을 요청한다고 해서  
매번 Root DNS, TLD DNS, Authoritative DNS를 모두 거치는 것이 아니라,  
ISP DNS에 캐시가 남아 있으면 바로 그 결과를 돌려줄 수 있다.

## 3. NAT는 왜 필요한가

SerahKim님 글에서 정리해주신 흐름을 다시 떠올려보면,  
기본 게이트웨이까지 도달한 뒤에는 공유기가 NAT를 수행한다.

NAT(Network Address Translation)는 내부 네트워크의 사설 IP를 외부 통신 가능한 공인 IP로 변환해주는 기술이다.

예를 들어 내부 요청은 아래처럼 바뀐다.

```text
192.168.0.3:50000
-> 공유기 NAT/PAT
-> 1.2.3.4:60000
-> 서버 123.45.67.89:80
```

여기서 중요한 점은 단순히 IP만 바꾸는 것이 아니라,  
여러 내부 기기를 구분하기 위해 포트까지 함께 매핑하는 `PAT(Port Address Translation)`가 자주 사용된다는 점이다.

이 과정을 통해 하나의 공인 IP로도 여러 내부 장치가 동시에 인터넷을 사용할 수 있다.

## 4. 이번에 연결해서 이해한 전체 흐름

이번에 다시 정리하면서 내가 이해한 실제 요청 흐름은 아래와 같다.

```text
1. 사용자가 www.google.com 입력
2. DNS 서버에 질의
3. 캐시와 TTL 확인
4. 필요하면 상위 DNS 서버를 따라가며 IP 조회
5. 목적지 IP 확인
6. 라우팅 테이블 확인
7. 외부 목적지라면 기본 게이트웨이 선택
8. ARP로 기본 게이트웨이의 MAC 주소 조회
9. 공유기로 프레임 전달
10. 공유기가 NAT/PAT 수행
11. 공인 IP 기반으로 인터넷 구간 통과
12. 최종 서버 도착
```

이렇게 보니 이전보다 각 개념이 더 선명하게 연결되었다.

- DNS는 이름을 IP로 바꾸는 단계
- Routing은 어디로 보낼지 결정하는 단계
- ARP는 실제로 누구에게 넘길지 MAC 주소를 알아내는 단계
- NAT는 내부 주소를 외부 통신 가능한 주소로 변환하는 단계
