# 모두의 연구소 LLM 6기 재직자 과정 4주차 3일차 정리

## 4주차 3일차

4주차 3일차는 검색을 평가하는 데서 한 걸음 더 나아가,  
검색 자체를 어떤 방식으로 구현할 수 있는지 직접 비교해보는 시간이었다.

이번에는 키워드 검색과 벡터 검색을 각각 구현해 보고,  
두 결과가 얼마나 겹치는지 확인한 뒤,  
TF-IDF와 BM25 같은 전통적인 텍스트 검색 점수,  
그리고 마지막에는 하이브리드 검색까지 연결했다.

이번 수업의 큰 흐름을 한 줄로 정리하면 아래와 같다.

```text
documents 준비
-> 임베딩 생성
-> keyword_search 구현
-> vector_search 구현
-> overlap_rate로 결과 비교
-> TF-IDF 점수 이해
-> BM25로 TF-IDF 한계 보완
-> keyword + vector를 합친 hybrid search로 확장
```

이번 내용을 통해 느낀 건,  
검색은 벡터 검색 하나만 알면 되는 게 아니라  
전통적인 keyword 기반 검색과 그 장단점까지 함께 이해해야 훨씬 입체적으로 보인다는 점이었다.

---

### Keyword Search와 Vector Search

> 1. 실습용 documents

이번 노트북의 문서셋은 Python, RAG, FAISS, 임베딩, 트랜스포머 같은 주제를 한두 문장으로 설명한 짧은 텍스트들이었다.

```python
documents = [
    "Python은 데이터 과학과 머신러닝에 널리 사용되는 프로그래밍 언어입니다.",
    "자연어 처리(NLP)는 컴퓨터가 인간의 언어를 이해하고 생성하는 기술입니다.",
    "벡터 데이터베이스는 고차원 벡터를 효율적으로 저장하고 검색하는 시스템입니다.",
    ...
]
```

데이터셋이 단순해서 오히려 좋았던 점은,  
각 검색 방식이 어떤 문서를 왜 상위에 올렸는지를 눈으로 확인하기 쉽다는 것이었다.

> 1. keyword_search

키워드 검색은 query와 문서를 공백 기준으로 나눈 뒤,  
겹치는 단어 수를 점수로 사용하는 매우 단순한 방식이었다.

```python
def keyword_search(query, docs, top_k=3):
    query_tokens = set(query.lower().split())
    scores = []
    for i , doc in enumerate(docs):
        doc_tokens = set(doc.lower().split())
        overlap = len(query_tokens & doc_tokens)
        scores.append((i, overlap))
```

이 방식은 구현이 정말 직관적이다.

- 같은 단어가 있으면 점수가 올라가고
- 없으면 점수가 오르지 않는다

즉 "정확한 용어 매칭"에는 강하다.

하지만 한계도 바로 보였다.

- 표현이 조금만 달라져도 놓칠 수 있고
- 동의어나 문맥 차이를 반영하지 못하고
- 한국어에서는 공백 단위 split만으로는 꽤 거칠다

즉 keyword search는 쉽고 빠르지만,  
표현 차이에 약한 검색 방식이라는 걸 다시 느꼈다.

> 1. vector_search

반대로 벡터 검색은 query와 문서를 임베딩한 뒤  
벡터 유사도를 계산해서 상위 문서를 찾는 방식이었다.

```python
def vector_search(query, docs, doc_embs, top_k=3):
    q_emb = np.array(embeddings.embed_query(query))
    similarities = np.dot(doc_embs, q_emb) / (
        np.linalg.norm(doc_embs, axis=1) *
        np.linalg.norm(doc_embs, axis=1) *
        np.linalg.norm(q_emb)
    )
```

코드상으로는 유사도 계산식이 들어가지만,  
개념적으로는 "문장을 의미 벡터로 바꿔서 비슷한 문장을 찾는다"로 이해하면 편했다.

이 방식의 장점은 분명했다.

- 단어가 완전히 같지 않아도 의미가 비슷하면 찾을 수 있고
- query와 문서 사이의 표현 차이를 조금 더 유연하게 다룰 수 있다

즉 벡터 검색은  
"단어"보다 "의미"에 더 가까운 검색 방식이라고 느껴졌다.

---

### 두 검색 결과 비교

> 1. 같은 query라도 결과는 다를 수 있다

노트북에서는 `"Python 프로그래밍 언어"`, `"딥러닝 모델"`, `"FAISS"` 같은 질의를 넣어서  
키워드 검색과 벡터 검색 결과를 비교했다.

여기서 느낀 점은 검색 방식마다 잘하는 문제가 조금 다르다는 것이었다.

- 고유명사나 정확한 용어는 keyword 검색이 강할 수 있고
- 표현 차이가 있거나 의미가 더 중요한 질의는 vector 검색이 유리할 수 있다

즉 어떤 방법이 절대적으로 더 좋다기보다  
질문 성격에 따라 장단점이 갈린다는 점이 중요했다.

> 1. overlap_rate

두 검색 결과가 얼마나 비슷한지 보기 위해 `overlap_rate`도 계산했다.

```python
def overlap_rate(keyword_results, vector_results):
    kw_ids = set(idx for idx, _ in keyword_results)
    vec_ids = set(idx for idx, _ in vector_results)

    overlap = kw_ids & vec_ids
    union = kw_ids | vec_ids
    overlap_rate = len(overlap) / len(union)
    return overlap_rate
```

이 값은 결국 두 결과 집합의 자카드 유사도와 비슷하다.

- 1.0에 가까우면 두 검색 결과가 거의 같고
- 0에 가까우면 서로 다른 문서를 많이 뽑았다는 뜻이다

이런 비교가 좋았던 건  
막연히 "벡터 검색이 더 좋아 보인다"가 아니라  
실제로 두 방식이 얼마나 같은 문서를 고르는지 객관적으로 볼 수 있었기 때문이다.

### TF-IDF

> 1. 단어 중요도를 수치로 보는 방식

수업 중간부터는 TF-IDF 개념을 다시 정리했다.

핵심은 두 요소였다.

- TF: 한 문서 안에서 특정 단어가 얼마나 자주 나오는가
- IDF: 그 단어가 전체 문서 집합에서는 얼마나 희귀한가

즉 TF-IDF는  
"이 문서 안에서 자주 나오고, 전체 문서에서는 흔하지 않은 단어"를 중요하게 본다.

실습에서는 직접 클래스로 구현해서 점수를 계산했다.

```python
class TFIDF:
    def __init__(self, documents):
        self.docs = documents
        self.tokenized = [doc.lower().split() for doc in documents]
        self.N = len(documents)
        self.df = {}
```

직접 구현해보니,  
전통적인 정보 검색 점수도 결국 토큰화, 문서 빈도, 단어 빈도를 조합하는 구조라는 점이 더 와닿았다.

> 1. TF-IDF의 한계

이번에 특히 기억에 남은 건 TF-IDF의 한계였다.

1. 단어가 많이 나올수록 점수가 계속 선형으로 증가한다.
2. 문서 길이 차이를 충분히 자연스럽게 보정하지 못한다.

즉 어떤 단어가 10번 나왔든 100번 나왔든  
실제 관련성은 그렇게까지 차이가 나지 않을 수 있는데  
TF-IDF는 이걸 너무 단순하게 다룬다.

또 긴 문서에서 우연히 단어가 몇 번 나온 경우와  
짧은 문서에서 핵심적으로 나온 경우를 구분하는 데도 한계가 있다.

---

### BM25

> 1. TF-IDF를 더 현실적으로 보완한 방식

BM25는 TF-IDF 아이디어를 유지하면서도  
검색 품질에 더 맞게 보정한 방식으로 소개되었다.

핵심은 아래 두 가지였다.

- 단어가 많이 나온다고 점수가 무한정 커지지 않게 한다
- 문서 길이에 따라 점수를 조절한다

실습에서는 직접 BM25 클래스를 구현했다.

```python
class BM25:
    def __init__(self, documents, k1=1.5, b=0.75):
        self.k1, self.b = k1, b
        self.docs = documents
        self.tokenized = [doc.lower().split() for doc in documents]
        self.N = len(documents)
        self.avgdl = sum(len(tokens) for tokens in self.tokenized) / len(self.tokenized)
```

수식은 처음 보면 복잡하지만, 해석은 비교적 명확했다.

- TF 포화 효과를 주고
- 길이 보정을 하고
- 희귀 단어에 더 높은 가중치를 준다

즉 BM25는  
"단어가 얼마나 많이, 얼마나 의미 있게 등장했는가"를  
TF-IDF보다 더 현실적으로 보는 점수라고 이해하게 되었다.

> 1. 직접 구현할 때 주의할 점

이날 코드에서 함께 확인한 포인트도 좋았다.

- `score()` 마지막에 `return total`이 있어야 하고
- query를 문자 단위가 아니라 단어 단위로 순회해야 하고
- 토큰화가 이상하면 BM25도 그대로 이상하게 동작한다

즉 수식만 외우는 게 아니라  
실제로 코드로 옮길 때 어디서 실수하는지도 같이 보는 게 도움이 됐다.

### LangChain Retriever와 Hybrid Search

> 1. BM25Retriever, Vector Retriever

후반부에는 직접 구현한 검색 로직을 넘어서  
LangChain의 `BM25Retriever`와 FAISS 기반 vector retriever도 함께 사용했다.

```python
bm25_retriever = BM25Retriever.from_documents(docs_lc)
vector_store = FAISS.from_documents(docs_lc, embeddings)
vector_retriever = vector_store.as_retriever(search_kwargs={"k":3})
```

직접 구현한 것과 라이브러리 기반 도구를 같이 보면서,  
기초 개념을 알고 나면 프레임워크 도구가 훨씬 덜 블랙박스처럼 느껴진다는 점이 좋았다.

> 1. simple_hybrid와 EnsembleRetriever

마지막에는 keyword/BM25와 vector 검색을 섞는 하이브리드 검색도 다뤘다.

간단한 버전은 점수를 정규화해서 더하는 방식이었고,

```python
combined[idx] = kw_score + vec_score
```

LangChain 쪽에서는 `EnsembleRetriever`로 가중치를 조정하는 실습도 있었다.

```python
ensemble = EnsembleRetriever(
    retrievers=[vector_retriever, bm25_retriever],
    weights=[0.5, 0.5]
)
```

이 부분이 특히 흥미로웠던 이유는,  
검색을 하나의 정답 방식으로 보지 않고

- 정확한 키워드 일치
- 의미 기반 유사성

을 함께 반영하는 방향으로 생각하게 해줬기 때문이다.

실무에서도 keyword/BM25와 vector를 같이 쓰는 이유가 조금 더 납득됐다.

## 정리

4주차 3일차는 다양한 검색 방식을 나란히 놓고 비교해본 날이었다.

특히 아래 내용이 핵심으로 남았다.

1. keyword search는 정확한 용어 매칭에 강하지만 표현 차이에 약하다.
2. vector search는 의미 유사성에 강하지만 고유명사나 정확한 표기 일치에서는 keyword 방식이 더 유리할 수도 있다.
3. TF-IDF와 BM25는 전통적인 검색 점수의 핵심 개념이고, BM25는 TF-IDF의 한계를 더 현실적으로 보완한다.
4. hybrid search는 keyword와 vector의 장점을 함께 가져가려는 전략이다.

이번 내용을 통해  
RAG에서 retrieval을 개선한다는 말이 단순히 임베딩 모델만 바꾸는 것이 아니라,

- 어떤 검색 방식을 쓸지
- 점수를 어떻게 계산할지
- keyword와 vector를 어떻게 조합할지

까지 포함하는 훨씬 넓은 문제라는 걸 조금 더 체감하게 되었다.
