## 배경 및 기술 선택의 이유

가변적인 개수의 대용량 영상을 웹 브라우저에서 동시 재생하고 제어해야 하는 요구사항이 있었습니다.

기존 단일 MP4 방식은 전체 파일을 로딩해야 하므로 여러 영상을 동시에 불러올 때 초기 지연과 버퍼링이 심각했습니다.

이를 해결하기 위해 영상을 세그먼트 단위로 전송하는 HLS 프로토콜을 채택했습니다.

## 기술적 도전과 해결 과정

### 1. 변환 방식 비교와 선택

변환 속도가 빠른 Codec Copy 방식과 정밀한 제어가 가능한 Re-encode 방식을 비교 실험했습니다.

분석 결과 Codec Copy는 원본의 불규칙한 GOP 간격 때문에 Seek 정확도가 떨어지고 세그먼트 크기가 커서 버퍼링이 발생했습니다. 반면 Re-encode는 변환 시간은 길지만 세그먼트 크기를 크게 줄이고 GOP 경계를 맞출 수 있어 동기화 재생에 유리했습니다.

### 2. Semaphore 기반 동시 처리와 구조 개선

기존 Python Worker의 CPU 부담과 동시 처리 제한 문제를 줄이기 위해 Java 기반 서비스로 리팩터링했습니다.

Semaphore를 도입해 서버 자원에 맞게 동시 인코딩 개수를 제한했고, FFmpeg 옵션을 조정해 세그먼트 경계와 키프레임을 강제로 정렬했습니다.

### 3. 인프라 비용과 운영 효율

클라우드 미디어 변환 서비스와 자체 서버 구성을 함께 검토해 비용과 운영 효율을 비교했고, 서비스 규모와 예산에 맞는 현실적인 아키텍처를 도출했습니다.

## 성과

GB 단위의 고화질 영상 5개를 동시에 불러오고, 10초 전후 이동이나 재생/정지 같은 제어를 1초 미만 지연으로 수행하는 동기화 플레이어 환경을 구축했습니다.

단순 기술 도입이 아니라 성능 테스트 데이터를 근거로 인코딩 전략을 수립하는 미디어 백엔드 역량을 얻었습니다.
