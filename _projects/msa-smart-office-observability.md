## 기술적 페인 포인트

8명의 개발자가 가변적인 서비스를 함께 개발하면서 로컬 설정이 배포 환경에 그대로 반영되지 않아 런타임 에러가 반복되었습니다.

특히 서비스 간 호출에서 어느 지점에서 지연이나 실패가 발생하는지 알 수 없는 분산 시스템의 비가시성이 가장 큰 문제였습니다.

## 설계 방향

### 1. TraceId 기반 공통 로그 모듈

모든 요청의 인입부터 종료까지 TraceId를 생성하고 `response_time`을 자동 계산하는 AOP 공통 모듈을 직접 설계했습니다.

8명의 팀원이 즉시 적용할 수 있도록 JitPack 기반 라이브러리 형태로 배포했고, 이를 통해 모든 서비스가 동일한 형식의 로그를 남기게 만들었습니다.

### 2. ELK 파이프라인 구축

Filebeat, Logstash, Elasticsearch 파이프라인을 구성해 분산된 컨테이너 로그를 중앙 집계했습니다.

검증 과정에서 컨테이너 간 타임존 불일치로 로그 순서가 뒤섞이는 문제를 발견했고, Docker 환경의 TZ를 동기화하고 Logstash 필터를 조정해 Kibana에서 1ms 단위 흐름 추적이 가능하도록 만들었습니다.

### 3. Config Server와 CI/CD 표준화

Spring Cloud Config와 GitHub Actions를 이용해 모든 민감 정보와 환경 변수를 중앙 관리하고, 어떤 서비스든 동일한 배포 파이프라인을 타도록 구조화했습니다.

그 결과 환경 변수 누락으로 인한 배포 실패율을 0%까지 낮출 수 있었습니다.

## 성과

장애 발생 시 여러 서버 로그를 수동 탐색하던 방식을 TraceId 기반 중앙 검색으로 바꾸면서 평균 장애 분석 시간(MTTR)을 80% 이상 단축했습니다.

개별 개발자가 인프라 설정이 아니라 비즈니스 로직에 집중할 수 있도록, 신뢰 가능한 공통 인프라 레이어를 제공한 경험이었습니다.
